## 3. 各項擴充規格細節

### E-01 任選優惠（折扣 / 贈品）

#### 3.1.1 功能描述
商家可建立「任選 N 件指定範圍商品享 X 折／固定金額折扣／加贈贈品」的活動。例：「任選 3 件指定區商品，第 3 件免費」「任選 5 件 8 折」。

#### 3.1.2 計算位置
- 任選**折扣**：屬規則 §3.2 促銷折扣優惠的第 4 子類型，計算位置在 [優惠計算引擎 §Step 2](Evomni_優惠計算引擎_技術規格_PRD.md)
- 任選**贈品**：屬規則 §3.4 贈品優惠的子類型，計算位置在 [優惠計算引擎 §Step 7](Evomni_優惠計算引擎_技術規格_PRD.md)
- 適用互斥規則：與其他促銷折扣同一商品互斥（依優先序 1→6）
- 計入全店門檻：商品金額與件數均計入

#### 3.1.3 後台欄位（建議）
| 欄位 | 元件 | 說明 |
|------|------|------|
| 活動類型 | radio | 任選折扣 / 任選贈品 |
| 適用商品範圍 | 商品/分類選擇器 | 商家自訂任選池 |
| 任選件數 | input-number | 必填，最少 2 件 |
| 任選類型 | radio | 同款限選一件 / 不限同款 |
| 優惠內容 | input | 折扣率／固定金額／贈品商品 |
| 上限件數 | input-number | 每筆訂單最多套用幾組 |

#### 3.1.4 邊界情境
1. 任選池內商品同時觸發其他促銷折扣 → 依規則互斥優先序處理
2. 任選池內商品不足任選件數 → 不觸發
3. 退貨時退掉任選池內商品 → 視為任選未成立，是否解除折扣？**沿用 Part4 v1.2 §6.2J「加購退款」邊界處理原則：不重算 + 退款單揭露**

---

### E-02 定期購折扣 / 定期購贈品

#### 3.2.1 功能描述
針對訂閱式商品（例：保健品、清潔用品定期配送），提供「訂閱者專屬折扣」與「訂閱滿 N 期送贈品」功能。

#### 3.2.2 前置依賴
- **訂閱模組（暫無 PRD）**：需先建立 `subscriptions`、`subscription_orders` 資料表結構與 UI
- 訂閱模組未啟動前，本擴充項無法執行
- 建議與訂閱模組同步規劃，避免重複設計優惠計算邏輯

#### 3.2.3 計算位置
- 定期購**折扣**：規則 §3.2 促銷折扣優惠的**第 1（最高）優先序**子類型
- 定期購**贈品**：規則 §3.4 贈品優惠子類型，**送出時機固定不可自訂**

#### 3.2.4 待擴充規格啟動時定義之題目
1. 訂閱期數的「累計門檻」如何計算？跨年度是否歸零？
2. 訂閱取消時，已享過的定期購折扣是否追討？
3. 訂閱者轉換為一般訂單時，定期購折扣立即停止 vs 該期仍享用？

---

### E-03 點數兌換贈品

#### 3.3.1 功能描述
會員可於商品頁／購物車以點數兌換指定贈品（例：500 點兌換限量贈品包）。

#### 3.3.2 計算位置
- 屬規則 §3.1 Step 1（加購優惠子類型）
- 兌換的贈品**不適用任何折扣**
- 兌換贈品金額**不計入**全店滿額／滿件門檻

#### 3.3.3 後台欄位（建議）
| 欄位 | 元件 | 說明 |
|------|------|------|
| 兌換商品 | 商品選擇器 | 必填 |
| 兌換點數 | input-number | 必填 |
| 兌換上限 | input-number | 每人每月最多兌換次數 |
| 庫存設定 | input-number | 兌換贈品專屬庫存 |
| 活動期間 | date-picker | — |

#### 3.3.4 與既有功能差異
- vs 加購品：加購品需「付錢」，兌換贈品「付點數」
- vs 點數折抵：點數折抵是「扣款項」，點數兌換是「換實物」
- 兩者可同時觸發：會員既可用點數兌贈品，也可用點數折抵金額

### E-04 贈品共用上限「排程切換」

#### 3.4.1 功能描述
擴充 Part4 v1.2 §6.2F 的「贈品共用上限」設定，新增「排程切換」能力。

例：商家設定「2026-11-11 00:00 ~ 2026-11-13 23:59 期間共用上限改為 15」，雙 11 期間自動放寬、過後自動回復。

#### 3.4.2 後台欄位（建議）
- 行銷設定頁新增「排程清單」區塊
- 每筆排程：起始時間 / 結束時間 / 共用上限值 / 啟用開關
- 排程與排程不可重疊（驗證提示）

#### 3.4.3 前台 / 後台對應顯示變化
- 後台活動列表「贈品上限」欄位需加註「目前生效共用上限：N（因 OO 排程套用）」
- 排程結束自動回復為預設共用上限，無需人工切換

#### 3.4.4 設計取向待定（開發前與工程師討論確認）

**取向 A — 純後台手動排程**
- 系統行為：商家自行新增／編輯／刪除排程
- 工時：小
- 適合：商家清楚自己的促銷檔期

**取向 B — 整合 Evomni 行事曆**
- 系統行為：若 Evomni 未來有「促銷檔期行事曆」，排程可從行事曆自動帶入
- 工時：中
- 適合：商家有多個促銷檔期需統一管理

**待定案題目**：
1. 是否需整合行事曆？
2. 排程數量上限？

### E-05 Mode B（會員回饋負餘額）正式實作

#### 3.5.1 功能描述
擴充 Part4 v1.2 §6.2I 已定義的 Mode B 開通設定 UI，**完成 Mode B 後端實作**（v1.2 僅實作 Mode A）。

#### 3.5.2 v1.2 已具備
- 商家自助開通 toggle（後台 UI）
- 開通／關閉的 audit log
- 二次確認彈窗

#### 3.5.3 v1.3 需補的部分
| 項目 | 說明 |
|------|------|
| 會員錢包負餘額 schema 改動 | `members.point_balance`、`members.wallet_balance` 欄位需放寬至允許負值 |
| Mode A → Mode B 切換轉換邏輯 | 既有「應收記錄」自動轉為負餘額 |
| Mode B → Mode A 切換轉換邏輯 | 既有負餘額自動轉回「應收記錄」 |
| 會員前台顯示負餘額狀態 | 個人中心新增負餘額警示與還款引導 |
| 對帳報表新增負餘額分欄 | 商家對帳可看到當前負餘額會員總數 |

#### 3.5.4 待 Ray schema 評估結果決定
- 若 schema 改動成本過高，考慮以「獨立 `member_negative_balance` 表」記錄而非欄位改動
- 評估結論決定本項實作工時與啟動排程
