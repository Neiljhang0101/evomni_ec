## 三、客戶情境分類（最高決策標準）

本頁面需同時處理兩種客戶情境，UI 結構依情境有所不同：

| 情境代號 | 客戶類型 | 電商模組狀態 | 儀表板顯示策略 |
|---------|---------|------------|--------------|
| **情境 A** | 純電商 | 電商開通，無形象站 | 電商儀表板全區塊 + Sitemap 按鈕（次要樣式）；不顯示 CMS 統計區 |
| **情境 B** | 形象站 + 電商 | 兩者均開通 | 電商儀表板 + CMS 統計輔助區（含 Sitemap 按鈕，次要樣式）|
| **情境 C** | 純形象站 | 電商未開通 | 現有形象站儀表板行為，本文件不規格（另文件管轄）|

> **Sitemap 說明**：Sitemap 是所有網站類型（純電商 / 形象+電商）的共用 SEO 需求，兩種情境均需要此功能。現有設計的問題**只在視覺優先級**（全寬主要藍色按鈕），不是「要不要顯示」的問題。兩種情境均降格為次要操作，見 §六。

> **判斷依據**：依 `store.getters.ecommerce_enabled` 布林值決定情境 A/B；再依 `store.getters.cms_enabled` 決定是否顯示 CMS 統計輔助區。

## 四、現有設計 Skill 審核結論

> **審核對象**：形象站現有儀表板截圖（2026-05-21）
> **審核標準**：evomni-ux-spec 8 條強制規則 + DASHBOARD 頁型規格

### 4.1 違反 R1（頁面類型定義）— 🔴 必修

現有設計未套用 DASHBOARD 頁型規格：
- 無時間範圍切換器
- 無匯出按鈕
- KPI 卡片（今日銷售額 / 訂單數 / 新會員數）完全缺失
- 待處理快速入口（後台導覽架構規劃已定義）完全缺失

### 4.2 違反 R7（術語不得裸露）— 🟡 必修

現有設計中「**請務必完成 SMTP 的設定**」外露技術術語給使用者。

| 現有文字 | 應替換為 |
|---------|---------|
| SMTP 的設定 | 郵件寄送設定 |
| 系統通知信設定 | 寄件設定 |
| Webhook 管理 | 即時通知設定 |

### 4.3 違反 R8（空狀態三層格式）— 🟡 必修

「**此功能尚未完工，敬請期待**」是錯誤的空狀態設計：

**錯誤做法**（現有）：
```
[⚠ 橙色 icon]  統計數據
此功能尚未完工，敬請期待
```

**正確做法**（依 evomni-ux-spec 空狀態規範）：
- 功能尚未上線 → **不顯示此卡片位置**，用路由守衛或條件渲染隱藏
- 功能已上線但無數據 → 顯示：空狀態 icon（48px `ri-inbox-line` `#DCDFE6`）+ 一句白話說明 + 主 CTA
- 禁止以佔位卡片展示「敬請期待」，會讓使用者誤以為系統有問題

### 4.4 版面優先級問題 — 🟡 設計決策調整

現有「**更新 Sitemap**」為頁面最頂部全寬藍色主要按鈕。

**Skill 判斷**：Sitemap 更新是低頻維護操作（每次改完選單才需執行），不應佔據儀表板最高視覺優先級。**Sitemap 兩種情境均保留**，問題只有視覺層級。

修正方式（兩種情境統一）：
- 樣式降格：`<el-button size="small" plain>`，移除全寬藍色
- 位置移至儀表板右上角 PageHeader 操作區（與「匯出 CSV」並列），而非頁面最頂部獨佔一列

### 4.5 「AI Tokens 額度」「主機可用空間」放置位置問題 — ⚪ 提醒

這兩個資訊屬於「基礎設施狀態」，與電商業務數據性質不同。

**Skill 建議**：
- 保留在儀表板右欄顯示（對商家有參考意義）
- 改以「資訊卡」樣式而非「警示 icon」樣式，無異常時顯示正常數值，異常時才出現警告色
- 功能未完工期間：條件渲染隱藏，不顯示佔位

```
├─────────────────────────────────┬────────────────────────────┤
│ 熱銷商品 Top 5（左欄）           │ 最近訂單列表（右欄，5筆）   │
└─────────────────────────────────┴────────────────────────────┘
```

### 5.2 KPI 卡片規格

| 指標 | 數值說明 | 對比基準 | 空值處理 | 資料來源 |
|------|---------|---------|---------|---------|
| 今日銷售額 | 今日付款成功訂單的總金額（NTD，不含取消）| 昨日同期，顯示 ↑↓% | 顯示 `$0` | `orders` 表 |
| 今日訂單數 | 今日所有非取消訂單筆數 | 昨日同期，顯示 ↑↓% | 顯示 `0` | `orders` 表 |
| 今日新會員 | 今日完成驗證的新會員數 | 昨日同期 | 顯示 `0` | `members` 表 |
| 今日待出貨 | 今日 + 歷史所有已付款但尚未出貨的訂單筆數（跨日累計）| 昨日同期 | 顯示 `0` | `orders` 表 |

> **「今日待出貨」設計說明：**
> - 取代原「訂單轉換率 / 訂單完成率」指標。電商「轉換率」的市場標準定義為訪客轉購買率（典型值 1–5%），需外部工具（GA4）；系統原生可算的比率（已完成 ÷ 全部建立，典型值 75–90%）與市場定義量級差距大，放在儀表板會造成商家認知混淆。
> - 「今日待出貨」是商家每日最高優先行動項目，數字本身即行動觸發器，無需解釋「好壞標準」。
> - 完整轉換漏斗分析（商品頁 PV → 加購 → 結帳 → 付款）屬進階電商包「數據中心 > 行銷成效 > 購物車轉換漏斗」功能，在正確的位置呈現。
> - 「今日待出貨」為跨日累計，不只算今日建立的訂單，而是所有 `status = 'paid'` 且 `shipped_at IS NULL` 的訂單，確保商家不會漏看前幾天積壓的未出貨單。

**KPI 卡片 tooltip（R8 三層格式）**：

「今日待出貨」：
```
What：所有已付款但尚未出貨的訂單筆數（跨日累計，非僅今日）
Impact：此數字代表商家還有幾筆訂單需要安排出貨；持續累積代表出貨流程可能需要優化
Next Step：點擊前往訂單管理 > 待出貨列表
```

「今日銷售額」：
```
What：今日付款成功訂單的商品金額總計（不含運費、已扣折扣後）
Impact：反映今日實際收款能力
Next Step：趨勢持續下滑時，前往數據中心查看商品排行與行銷成效
```

### 5.3 待處理事項規格

- 每筆事項顯示：icon + 項目名稱 + 數字 badge + 「前往處理 >」文字連結
- 數字為 0 時：顯示「無待處理事項」（空狀態 icon + 綠色文字「一切正常」），不隱藏整個區塊
- 點擊連結行為：

| 事項 | 跳轉目標 |
|------|---------|
| 待出貨訂單 N 筆 | `/orders?tab=orders&status=paid` |
| 待審核退換貨 N 筆 | `/orders?tab=returns&status=pending` |
| 發票未開立 N 筆 | `/orders?tab=invoices&status=pending` |
| 庫存告警商品 N 筆 | `/products/inventory?alert=true` |

**快速工具列（待處理事項區塊底部）**：

```
─────────────────────────────────────────
快速入口：[訪客查詢訂單 ↗]（開新分頁，前往前台訪客訂單查詢頁）
```

> 設計說明：訪客查詢訂單為前台客戶自助功能，商家偶爾需要將此連結傳給未登入的購買者。以「快速入口」文字連結形式放在待處理事項底部，不與主要 action item 爭視覺優先級。

### 5.4 訂單趨勢圖規格

- 圖表類型：折線圖（Line Chart）
- 時間軸：X 軸為日期，Y 軸為訂單金額（NTD）
- 預設範圍：最近 7 天；可切換至 30 天、90 天
- 同時顯示「本期」與「上期同區間」兩條折線（對比色：品牌藍 vs 灰）
- 空狀態（開通首日無數據）：顯示「尚無銷售記錄，當您有第一筆訂單後，趨勢圖將自動呈現」+ 圖表區域顯示空白折線框（不顯示 ⚠ icon）

### 5.5 熱銷商品 Top 5 規格

| 欄位 | 說明 |
|------|------|
| 排名 | 1–5 |
| 商品縮圖 | 24×24px，圓角 4px |
| 商品名稱 | 可點擊，跳至商品編輯頁 |
| 銷量 | 本期（依時間範圍）已完成訂單的商品數量 |
| 銷售額 | NTD |

空狀態：顯示「尚無商品銷售記錄」+ 「前往建立商品」CTA 按鈕。

> **Q4 定案（連動時間範圍）**：熱銷商品 Top 5 的統計時間範圍跟隨全頁時間範圍選擇器，切換時重新請求 API（帶入 `date_from` / `date_to` 參數）。

### 5.6 系統初始化提示欄規格

**顯示條件**：電商開通後，系統自動偵測以下 4 個設定的完成狀態，任一未完成則顯示此欄。

```
背景：#FFFBE6，左側 3px 黃色邊條（#E6A817）
標題：「完成以下設定，即可開始接受訂單」
步驟條（橫向，4 步驟）：
  1 郵件寄送設定   2 金流設定   3 物流設定   4 建立第一個商品
（完成的步驟：綠色勾 icon + 灰色文字；未完成：數字圈 + 黑色粗體 + 「前往設定 >」文字連結）
```

**4 步驟設計依據**（對標市場主流電商平台）：
| 步驟 | 必要理由 | 缺少的後果 |
|------|---------|---------|
| 1 郵件寄送設定 | 訂單確認信、會員驗證信依賴此設定 | 客戶下單後收不到確認信，信任度崩潰 |
| 2 金流設定 | 沒有金流無法收款 | 無法完成任何線上訂單 |
| 3 物流設定 | 沒有物流無法出貨 | 訂單建立後無法選擇配送方式 |
| 4 建立第一個商品 | 沒有商品前台無法顯示任何東西 | 商家開站後看到空白前台 |

**隱藏條件**：4 個步驟全部完成後，自動隱藏（不可手動關閉）。
**注意**：步驟可以不按順序完成，系統逐項打勾即可。

### 5.11 載入 / 錯誤 / 重試狀態

**各區塊獨立載入**（非整頁 spinner），使用 Skeleton 骨架屏。

| 情境 | 行為 |
|------|------|
| 首次載入 | 各 widget 獨立顯示 Skeleton（灰色動態閃爍佔位），KPI 卡片寬度保持不變 |
| 單一 widget 載入失敗 | 該 widget 顯示「數據載入失敗 [重試]」（text 按鈕），其他 widget 正常顯示 |
| 整頁 API 失敗（500）| PageHeader 下方出現 `<el-alert type="error">` 橫幅：「數據服務暫時無法連線，請稍後重新整理頁面」|
| 網路離線 | 同整頁 API 失敗處理方式 |

## 六、情境 B — 形象站 + 電商儀表板規格

### 6.1 差異說明

情境 B 在情境 A 的基礎上，增加「CMS 統計輔助區」，並調整 Sitemap 更新按鈕位置。

> **Q5 定案（CMS 統計與電商 KPI 共用同一時間範圍選擇器）**
> 依工程師雙站架構規劃：形象站與電商站資料存於同一資料庫，以欄位辨識站台歸屬。因此 CMS PV/詢問單數與電商訂單數查詢的時間範圍條件可共用同一個查詢參數，無跨系統技術障礙。使用同一時間範圍選擇器亦能保持 UI 一致性，商家不必設定兩個不同的時間範圍。

## 七、「此功能尚未完工」現有卡片的處置策略

現有截圖中出現的「此功能尚未完工，敬請期待」卡片，採以下處置：

## 九、待確認事項與版本紀錄

### 9.1 決策事項紀錄（Q1–Q5 全數定案）

| 編號 | 問題 | 決策結果 | 定案依據 |
|------|------|---------|---------|
| Q1 | KPI 第 4 張指標如何選定？ | **定案**：「今日待出貨」= 所有 `status='paid'` 且 `shipped_at IS NULL` 的訂單數（跨日累積）；原「訂單完成率」因量級（75–90%）與市場轉換率標準（1–5%）差距大，易造成認知混淆而廢棄 | 今日待出貨為即時操作型指標，對日常出貨作業有直接行動意義 |
| Q2 | 初始化步驟 3 步或 4 步？ | **定案**：4 步驟（郵件 / 金流 / 物流 / 建立商品）| 缺少物流設定無法選擇配送方式，等同開張即殘缺；對標 Shopify/Shopline 均包含物流步驟 |
| Q3 | 儀表板是否加入「訪客查詢訂單」快速捷徑？ | **定案：加入**（待處理事項區塊底部文字連結）| 訪客無法自行查詢訂單需要客服介入，後台快速入口可降低操作摩擦；介面以低視覺權重文字連結呈現，不混淆主要操作 |
| Q4 | 熱銷商品 Top 5 時間範圍連動或固定本月？ | **定案**：連動全頁時間範圍選擇器 | 數據一致性更好，避免商家對不同時間範圍的數字感到困惑 |
| Q5 | 情境 B 的 CMS 統計是否與電商 KPI 共用時間範圍？ | **定案**：共用同一時間範圍選擇器 | 雙站共用同一資料庫（欄位辨識站台），無跨系統技術障礙；共用選擇器 UX 更一致 |
