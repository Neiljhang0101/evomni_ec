## 給 UI/UX 設計師的文件說明

在閱讀這份回饋整理之前，請先理解以下幾個重要的框架原則。這些原則決定了你在重新規劃時的思考方式，不只是逐條回應回饋。

### 原則一｜使用者回饋代表真實情境的體驗，不是功能清單

這份回饋來自設計師的真實操作體驗，每一條背後都代表一個實際情境下的卡點。在回應這些回饋時，不要只看表面的「需要什麼功能」，而是要問：**這個人在什麼情境下、做什麼任務、遇到了什麼阻礙？** 解決阻礙才是目標，不是把功能一條一條加上去。

### 原則二｜有些需求仍然需要整體性思考，不能只是逐條新增

部分回饋看似是「加一個功能就能解決」，但如果只是這樣處理，介面只會愈加愈複雜，最終變成另一個讓人困惑的系統。**每一個新功能的加入，都需要思考它對整體操作流程的影響。** 解決的是關鍵操作步驟，不是在現有介面上堆疊功能。

### 原則三｜這份回饋來自執行端（設計師），但使用者不只有設計師

設計師是系統的重度使用者，但選單管理同時也會被**客戶（網站管理者）**使用。客戶的技術能力更低、使用頻率更低、容錯能力更低。任何介面決策，都需要同時考量：
- 設計師（首次建立、複雜架構操作）
- 客戶（日常維護、局部調整）

不能只優化設計師的操作路徑，而忽略客戶的使用情境。

### 原則四｜此文件範圍僅限「選單的介面體驗」

本文件的回饋**只涵蓋選單管理的介面操作體驗**，不包含以下議題（這些需要另案討論並整體整合）：
- 麵包屑的生成邏輯與前台顯示
- 列表樣式 / 頁面設計的範本選擇
- 前台左側選單 / 上方選單的渲染邏輯
- 語系頁面的整體規劃

最終這些議題需要整體性地整合在一起，選單介面體驗只是其中一個環節。

### 原則五｜RWD 必須納入思考

後台選單管理介面目前只有桌面版設計。但客戶可能在平板甚至手機上進行日常維護操作。**在規劃任何介面元件時，請同步思考在窄螢幕下的操作方式**，特別是：
- 拖曳排序在觸控裝置上的替代方案
- 巢狀樹狀結構在小螢幕的折疊策略
- inline 新增子項目在行動裝置上的展開方式

### 原則六｜請以設計師面臨的客戶真實案例作為規劃基礎

選單管理最終是為了服務真實的網站架構需求。在重新規劃時，請參考設計師實際遭遇的客戶案例，例如：
- 電商客戶的多層產品分類
- 企業型客戶的多語系多層選單
- 內容型客戶的新聞分類加精選文章混合結構
- 隱私權政策、購物須知等不在主選單但需要獨立頁面的情況

**不要只設計「一般情況」的流程，要確保「邊緣情況」也有合理的處理方式。**

### 原則七｜遇到猶豫不確定的地方，務必尋求確認

選單管理牽涉前端渲染邏輯、後端 API 設計、客戶操作習慣等多個層面。**任何不確定的決策，都不要自行假設，務必與以下對象確認後再定案：**
- 使用者（設計師 / 客戶）：確認操作習慣和真實需求
- 技術端（前端 / 後端工程師）：確認技術可行性和實作成本
- PM：確認與其他模組的整合邏輯是否一致

| # | 問題描述 | 回饋來源 |

### C｜視覺與易用性（需優化）

| # | 問題描述 | 回饋來源 |

| C1 | 主選單層級與子層視覺太相近，架構複雜時難以辨識；需要層級間距差異或底色差異 | 02、03、04、05 |

## 四、UI/UX 需主動確認並推進的事項

以下三件事是目前的未定項，**由 UI/UX 負責發起討論、收集意見、提出建議方案並確認定案。** 未確認前請勿自行假設並繼續設計，需與使用者及技術端討論後再定案（見原則七）。

### D1｜語系複製規格相容性確認
語系複製功能在原本的選單規劃中已有定義，**不需要從零討論，但需要確認**。本次選單管理邏輯有重大更新（節點類型重新定義、群組新增、子層允許規則調整），請重新檢視：
- 原本的語系複製規劃，在新的節點邏輯下是否仍然適用？
- 自訂網址的語系路徑複製規則，是否需要因應新邏輯調整？
- 群組節點的「點擊行為」設定，複製到其他語系後的行為是否一致？

> 請先調出原有的語系複製規格文件，逐點對照本次更新的內容，確認有無需要調整的地方，再決定是否需要重新規劃。

### C4｜「群組」類型名稱
設計師普遍看不懂「群組」的用途，請訪談 2–3 位設計師及客戶後提出建議命名，幾個方向供參考：
- 「導覽群組」
- 「選單群組」
- 「分類入口」
- 保留「群組」但在類型下拉旁加說明文字（tooltip 或 hint）

> 這個命名不只影響後台介面，也影響客戶的理解。確認命名後需同步更新 PRD 所有相關段落。

### A3 / A5｜詞彙定義
兩個詞彙需要訪談確認後統一定案，並更新至 PRD 和所有介面文案：
- **「換頁」**：指「點擊後在同一視窗開啟（同 tab）」，但設計師誤解。建議改名為「同視窗開啟」，對應「另開瀏覽器分頁」，請確認後定案
- **「導覽標籤」**：設計師建議改為「選單名稱」，更直覺。請確認是否統一更名

## 六、UI/UX 主動盤點提醒

**設計師回饋只是冰山一角。沒有被提到，不代表沒有問題。**

這份回饋來自短期測試，設計師不一定覆蓋了所有操作情境。在回應這些回饋的同時，UI/UX 需要主動盤點以下面向，確認有無尚未被提出但同樣存在問題的操作體驗：

**操作流程完整性**
- 從零建立一個有三層結構的選單，完整操作流程是否順暢？
- 客戶日常維護（新增一個節點、調整順序、暫時隱藏）的操作路徑是否足夠簡單？
- 首次使用和熟練使用的操作路徑是否有做區分？

**錯誤與例外情境**
- 使用者操作錯誤時（例如加錯父層、刪錯節點），是否有足夠的撤銷或復原機制？
- 來源資料被刪除後，選單節點的警示是否夠清楚、修復路徑是否直覺？
- 儲存失敗時的提示與恢復流程是否完整？

**客戶視角的操作情境**
- 客戶（非設計師）第一次進入選單管理，是否能理解每個元件的用途？
- 客戶只需要改一個選單名稱，操作步驟是否足夠少？
- 客戶誤操作的風險點在哪裡？有沒有足夠的防呆？

**邊緣案例**
- 選單節點數量很多（例如 20 個主選單）時，介面是否仍然可用？
- 選單名稱很長時，節點 row 的顯示是否會破版？
- 多語系 + 多層結構同時存在時，操作複雜度是否在可接受範圍內？

> 建議在正式提交設計稿前，用真實的客戶案例（例如電商多層分類、多語系企業網站）走一次完整的操作流程，確認以上面向沒有明顯問題。遇到任何疑慮，回到原則七：與使用者及技術端討論確認。
