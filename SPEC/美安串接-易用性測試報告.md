## 三、問題清單（依嚴重程度排序）

### 🟠 Major — M1：API Access Key 編輯模式使用明文輸入框

- **影響任務：** T1（填入 API 憑證）
- **位置：** `html/美安串接設定.html` 第 283 行
  ```jsx
  <EInput value={accessKey} onChange={...} placeholder="請填入 API Access Key" width={400} />
  ```
- **問題描述：** 點擊「重新設定」後，API Access Key 的輸入框呈現為普通文字欄（`EInput`），使用者輸入時金鑰以明文顯示，旁人可直接看見。API Secret Key（第 297 行）則正確使用了 `PasswordInput`，造成同區塊兩欄不一致。
- **根本原因：** Access Key 的 edit mode 呼叫 `EInput`，未與 Secret Key 相同呼叫 `PasswordInput`。
- **建議修正：** 將第 283 行改為 `<PasswordInput>`，與 Secret Key 欄保持一致。同時補上顯示/隱藏切換功能（UX Spec Component List 要求）。

---

### 🟠 Major — M2：Screen 6 日期範圍選擇器實作不符規格

- **影響任務：** T6（匯出美安報表）
- **位置：** `html/美安串接設定.html` 第 484–487 行
  ```jsx
  <EInput type="date" value={dateFrom} onChange={setDateFrom} width={170} />
  <span style={{ color: '#C0C4CC' }}>—</span>
  <EInput type="date" value={dateTo}   onChange={setDateTo}   width={170} />
  ```
- **問題描述：** 匯出確認 Dialog 中日期篩選使用兩個獨立的日期輸入框，缺少「一鍵清空全部日期」的功能；兩欄之間無法驗證「結束日期不得早於起始日期」等基本邏輯。非專家使用者填完後可能不確定是否需要兩欄都填才算選定範圍。
- **根本原因：** 使用 `EInput type="date"` 替代 UX Spec 要求的 `el-date-picker type="daterange" clearable`。
- **建議修正：** 改為統一的日期區間選擇器元件（`daterange picker`），含清空（clearable）功能；或至少在目前兩欄方案上加入「清除全部」按鈕及輸入驗證。

---

### 🟠 Major — M3：Screen 5 手動傳送 Dialog 顯示不應出現的錯誤訊息

- **影響任務：** T5（傳送取消通知；間接）、最高權限功能
- **位置：** `html/美安串接設定.html` 第 721–724 行
  ```jsx
  <FormField
    label={'請輸入「確認」以繼續'}
    helper={'輸入與「確認」完全相符後才可操作'}
    error={confirmTxt && !isConfirmable ? '請輸入「確認」後才能執行此操作' : ''}
  >
  ```
- **問題描述：** 使用者開始輸入但尚未打完「確認」時，即立刻顯示紅色錯誤訊息「請輸入「確認」後才能執行此操作」。視覺上像是輸入錯誤，可能讓使用者以為填錯欄位。
- **根本原因：** 條件 `confirmTxt && !isConfirmable` 在輸入到一半時即為真，觸發 error 顯示。
- **UX Spec 要求：** 「未輸入「確認」就試圖操作 → 確認傳送按鈕保持 disabled（不顯示錯誤文字，靠 disabled 狀態傳達）」
- **建議修正：** 將 error prop 改為 `''`（或移除），僅依靠按鈕 disabled 狀態傳達限制。

---

### 🟡 Minor — N1：Prototype 無法展示初次設定的空欄位狀態

- **影響任務：** T1
- **位置：** `html/美安串接設定.html` 第 208–211 行
  ```jsx
  const [storeId, setStoreId] = React.useState('EVOMNI-TW-001');
  const [accessMasked, setAccessMasked] = React.useState(true);
  const [secretMasked, setSecretMasked] = React.useState(true);
  ```
- **問題描述：** Prototype 一進入即呈現「已設定並儲存」的遮罩狀態。真實的初次設定情境（三欄空白，商家尚未填任何 API 憑證）無法直接驗測。設計目標 1 要求計時「從進入設定頁到成功複製 XML URL」，但若無法模擬初始空白狀態，測試值偏低。
- **建議修正：** 加入「切換為初次設定狀態」的 Demo 按鈕，將 storeId 清空並取消遮罩，讓測試人員能驗測空白流程。

---

### 🟡 Minor — N2：訂單來源篩選標籤合併於 placeholder

- **影響任務：** T3
- **位置：** `html/美安串接設定.html` 第 415 行
  ```jsx
  placeholder="訂單來源：全部"
  ```
- **問題描述：** 篩選器以 `訂單來源：全部` 作為 placeholder，選擇後 placeholder 消失，使用者可能忘記這個欄位的用途。UX Spec 要求標籤（label）與 placeholder 分離：`Filter label: 訂單來源`（獨立顯示）、`Select placeholder: 全部`。
- **建議修正：** 在篩選器左側加上獨立的文字標籤「訂單來源」，Select placeholder 改為單純的「全部」。

---

### 🟡 Minor — N3：State B 傳送失敗錯誤訊息未使用 Alert 元件

- **影響任務：** T4（判讀傳送狀態）
- **位置：** `html/美安串接設定.html` 第 565–567 行
  ```jsx
  <span style={{ color: '#F56C6C', fontSize: 13 }}>API Error: Invalid store credentials (403)</span>
  ```
- **問題描述：** 錯誤訊息以 inline 紅字呈現，缺少左側色條、背景色等視覺容器，嚴重性較不明確。UX Spec 要求使用 `el-alert type="error"` Alert 元件，提供更清晰的視覺層級。
- **建議修正：** 改以 Alert 元件包裹錯誤訊息，增加背景底色與圖示。

---

### 🔵 Enhancement — E1：isSuperAdmin 固定為 true，無法驗測一般商家視圖

- **影響任務：** T4、T5 相關的最高權限流程
- **位置：** `html/美安串接設定.html` 第 505 行
  ```jsx
  const isSuperAdmin = true;
  ```
- **問題描述：** 最高權限功能區塊（查看傳送 LOG、手動傳送此訂單至美安）在 Prototype 中永遠顯示，無法模擬一般商家管理員的視圖，也無法驗測這些元素在非最高權限下是否完全不渲染。
- **建議修正：** 加入切換按鈕，讓測試人員可在「最高權限」與「一般管理員」兩種角色之間切換，驗測 v-if 隱藏行為。
