## 三、問題清單（依嚴重程度排序）

---

### P-01 — Token 驗證示範固定回傳失敗，無法體驗完整串接流程

**嚴重程度：** 🔴 Critical
**影響任務：** T1（首次申請串接）
**位置：** `html/飛信電子報串接設定.html:305`

```js
// 固定 1.5 秒後顯示失敗，無成功路徑
setTimeout(() => { setVerifying(false); setApiError(true); }, 1500);
```

**問題描述：** 非專家用戶完成申請步驟、貼入任意 Token 後，點擊「驗證並完成串接」，系統一律顯示「Token 驗證失敗」紅色錯誤訊息。用戶無法得知這是 Prototype 的示範限制，會以為是自己操作錯誤。唯一能進入已串接畫面的路徑是旁邊顯眼度低的「（示範）切換至已串接」按鈕，但此按鈕的存在本身也會讓用戶困惑（「這是什麼？我要點這個嗎？」）。

**根本原因：** 驗證邏輯沒有設計模擬成功路徑。

**建議修正：** 加入條件判斷：Token 長度 >= 32 字元則模擬驗證成功（`setApiError(false)` → 呼叫 `onSuccess()`）；長度 < 32 字元則維持現有失敗路徑。可在輸入框說明文字旁加一行灰色小字「（示範：貼入 32 字元以上的任意文字即可驗證通過）」。

### P-02 — 0 筆訂閱者防呆機制缺失

**嚴重程度：** 🔴 Critical
**影響任務：** T3（匯出訂閱者名單）
**位置：** `html/飛信電子報串接設定.html:732`

```jsx
<EvoBtn variant="secondary" onClick={() => setShowExportDialog(true)}>匯出至飛信</EvoBtn>
```

**問題描述：** 點擊「匯出至飛信」直接開啟確認 Dialog，不論訂閱者數量是否為 0。UX Spec Screen 3 明確規定「訂閱者 = 0 則顯示 warning message，不開啟 Dialog」。在 Prototype 示範情境中，SUBSCRIBERS 陣列有資料不會觸發，但工程師若依此實作則會缺少防呆。

**根本原因：** `onClick` handler 未包含訂閱者數量檢查邏輯。

**建議修正：** 在 onClick 中先檢查有效訂閱者數量（status === 'subscribed'）：若為 0，改呼叫 `showToast('目前沒有有效訂閱者可匯出', 'warning')`；大於 0 才開啟 Dialog。

---

### P-03 — 「前往飛信後台」按鈕位置與 Spec 不符

**嚴重程度：** 🟠 Major
**影響任務：** T2（確認狀態、前往後台）
**位置：** `html/飛信電子報串接設定.html:534-536`

**問題描述：** 按鈕被放在 PageHeader 右側（頁面最頂端），與狀態 Banner 在視覺上是分離的。UX Spec Screen 2 的 Information Hierarchy 明確標示「Primary CTA: 前往飛信後台」應在 Banner 一行，Component List 亦規定此按鈕為「suffix ArrowRight icon」的 el-button type="default"。非專家用戶的視線焦點會停留在 Banner 上，可能錯過位於頁面右上角的按鈕。

**根本原因：** 實作時將按鈕插入 PageHeader children，而非放置在 Banner 右側。

**建議修正：** 將「前往飛信後台」按鈕移至 StatusBanner 元件內部，靠右對齊，與狀態文字同行顯示。

---

### P-04 — 匯出紀錄表格欄位標題多加括號說明

**嚴重程度：** 🟡 Minor
**影響任務：** T5（查看匯出紀錄）
**位置：** `html/飛信電子報串接設定.html:591`

**問題描述：** Table 欄位標題顯示為「飛信群組名（匯出名單）」，而 UX Spec Screen 2 的 Column Headers 明確規定為「飛信群組名」。括號內的「匯出名單」說明在欄位值本身（如「2026-05-14 10:30 匯出」）已有足夠上下文，標題重複說明略顯冗餘，且與規格不符。

**根本原因：** 實作時主動補充說明文字。

**建議修正：** 將欄位標題改為「飛信群組名」。

---

### P-05 — Skeleton 載入狀態未實作

**嚴重程度：** 🟡 Minor
**影響任務：** 全部任務（進頁面初始）
**位置：** Screen 1、Screen 2 整體

**問題描述：** UX Spec Screen 1 和 Screen 2 均規定「Loading（頁面初始）」狀態顯示 Skeleton 取代 Banner 與說明區塊，但 Prototype 直接渲染完整畫面，無 Skeleton 過渡。在 Prototype 中可接受，但需提醒工程師實作，否則首次進入頁面時 API 載入延遲會造成畫面閃爍或空白。

**根本原因：** Prototype 不模擬 API 載入延遲，Skeleton 元件未建立。

**建議修正：** 工程師實作時依 Spec 於頁面初始顯示 Skeleton，直到後端 API（設定資料 + Token 靜默驗證）回應後才渲染實際內容。

### P-06 — 確認匯出 Dialog 群組命名文字顏色偏淺

**嚴重程度：** 🟡 Minor
**影響任務：** T3（匯出名單）
**位置：** `html/飛信電子報串接設定.html:800`

```jsx
<p style={{ color: '#909399' }}>
  在飛信系統中，此批名單將以「2026-05-14 10:30 匯出」命名儲存。
</p>
```

**問題描述：** 群組命名說明文字使用 `#909399`（placeholder 色），比 UX Spec 規定的 `#606266` 更淺。群組名稱是讓商家與發信同事溝通用的重要資訊，過淺的顏色可能讓用戶忽視這個關鍵說明。

**根本原因：** 實作時使用了更淺的 placeholder 色。

**建議修正：** 改為 `color: '#606266'`。

## 五、受測者語錄（模擬非專家用戶）

以下語錄為 AI 根據實際 UI 文字與互動流程推估，代表不熟悉後台系統的用戶在操作時可能說出的話：

1. 「這個『開始使用飛信電子報』我按下去了，然後它說申請成功，可是下面那個框框讓我貼 Token，我的 Token 是什麼？我要去哪裡找？步驟寫了，但我不確定『帳號管理 > API 管理』是在飛信的哪裡。」

2. 「我貼了 Token 按驗證，它說失敗，但我明明是複製貼上的，頭尾也沒有空格……這個旁邊有個很小的按鈕說『示範切換至已串接』，這個是什麼意思？我應該按嗎？」

3. 「頁面頂端有寫串接成功，是綠色的，我看到了。但我要去飛信後台，那個按鈕在哪裡？（往下找）喔，是右上角那個，我以為那個是設定的按鈕。」

4. 「我要匯出名單，那個『匯出至飛信』按鈕在工具列這裡……確認匯出的視窗有寫『2026-05-14 10:30 匯出』，這個是說我的名單在飛信那邊會叫這個名字嗎？那我記一下，這樣我才能叫同事找到它。」

5. 「（看到 Token 失效橘色 Banner）這個是說我的 Token 壞掉了？所以我要重新設定？我點了『重新設定 Token』，它問我確認，好，我就按繼續。（被帶回 Screen 1 的 Token 輸入框）所以我要重新去飛信拿 Token 嗎？那步驟在這裡嗎？（往上找）找不到步驟……」

6. 「停用之後飛信的名單會不會被刪掉？（讀 Dialog）哦，帳號與名單不受影響，好，那我就放心停用了。」

7. 「下面這個表格，可以看到上次匯出是 5 月 14 日，256 筆，成功。5 月 10 日那次有部分失敗……我點『查看詳情』，可以看到失敗的是哪些地址，這個不錯。」
