## Part 1 — Design Brief

### Product Vision

會員中心是消費者完成購買後最常回訪的自助入口，涵蓋訂單查詢、個人資料管理、點數與優惠券使用、帳號安全設定等 9 個功能頁面。設計目標是讓消費者不需要聯絡客服，就能自己完成所有帳號相關操作，同時透過點數和優惠券頁面強化回購意願。頁面架構以「我的訂單」為預設入口，左側導覽（桌機）或底部 Tab（手機）提供一致的功能導覽。

### Target User

**主要用戶：** 完成購買後的消費者，希望追蹤訂單狀態、管理收件地址、使用積累的點數與優惠券；多以手機瀏覽，期望操作路徑短。

**次要用戶：** 有多組身份驗證需求的消費者（LINE/Google 登入）或有安全顧慮的消費者，需要管理第三方帳號綁定與登入裝置。

### Design Goals

| # | 目標 | 驗證方式 |
|---|------|---------|
| 1 | 消費者從「我的訂單」找到特定訂單並查看物流狀態，全程不超過 30 秒 | 計時觀察從進入訂單列表到看到物流追蹤資訊 |
| 2 | 消費者能在點數頁一眼看到「可用點數」與「即將到期」，不需要翻頁或計算 | 觀察消費者第一眼注視的位置，詢問到期資訊 |
| 3 | 消費者在新增地址表單中，能不出錯地完成縣市/行政區選擇 | 觀察消費者完成地址新增所需的嘗試次數 |
| 4 | 消費者能在帳號安全頁識別「哪些裝置不是我的」並完成遠端登出 | 觀察消費者是否能在 60 秒內找到並點擊「登出此裝置」 |
| 5 | 手機版底部 Tab 讓消費者切換任一主要功能不超過 2 次點擊 | 計算從主畫面切換至目標功能所需的點擊次數 |

### Design Principles

1. **自助優先，減少客服接觸** — 所有常見帳號操作（查單、改地址、改密碼）都能在頁面內完成，不引導消費者去找客服
2. **資料呈現清楚，不混淆** — 點數、訂單金額、折扣明細各自獨立呈現，不讓消費者自己去心算
3. **危險操作要有防線** — 刪除地址、刪除帳號、登出所有裝置等不可逆操作，一律需要確認步驟
4. **RWD 體驗等質** — 桌機與手機的功能完整度相同，僅佈局與導覽方式不同，不在手機版縮減功能

### Design System Reference

- **設計系統：** Evomni 前台設計系統（`frontend-design-system/`）
- **字體：** Inter（拉丁文字）+ Noto Sans TC（中文），基礎字級 16px，最小字級 14px
- **圖示：** Material Icons Outlined（CDN）
- **色彩：** CSS 自訂屬性，引用 `colors_and_type.css`
- **彈窗 / 通知：** EvomniAlert（`evomnialert/`）— 所有確認對話、操作回饋均使用此元件，禁止自製 modal 或 toast
- **頁面背景：** `var(--fill-300)`（米白）
- **頁面最大寬度：** `var(--container)`

### Accessibility Requirements

- **Target WCAG level:** AA
- 所有文字最小 14px
- 狀態 Tag 需同時提供顏色與文字（不能只靠顏色區分狀態）
- 表單欄位 focus 狀態需清楚顯示（輪廓明顯）
- 手機觸控按鈕最小 36×36px，操作元素之間留足間距
- 訂單狀態時間軸需提供可讀文字，不能只靠視覺進度呈現

### Hard Constraints

- 禁止使用 Emoji 字元（所有文案、標籤、說明文字）
- 前台頁面不使用後台設計系統（Element Plus、`components/` 資料夾元件）
- 所有彈窗、確認對話、通知訊息一律使用 EvomniAlert，禁止自製
- 英文縮寫需附中文說明（例：「成交金額（GMV）」）；CTA 按鈕英文全大寫例外（如 MY ACCOUNT）
- 會員端不顯示任何應收/點數註銷/後台方案名稱（啟航方案、進階電商包）等內部資訊
- 心願清單導覽項目：由後端 `wishlist_enabled` 控制，未啟用時前端完全不渲染此項目，路由回 404 並重導向訂單頁

### Out of Scope

- 退換貨申請表單（由 Part 3 訂單管理負責，本文件只提供入口按鈕）
- 外部 OAuth 授權頁面（LINE / Google / Facebook 各自的授權頁）
- 電子發票 API 串接細節
- Mode B（負餘額點數）完整 UI（待後端 v1.3 啟用後補齊）
- 推薦好友功能的後端推薦連結生成邏輯

### Screen 1：整體佈局框架

**Purpose：** 提供會員中心的全局框架，包含頂部會員資訊卡、左側功能導覽（桌機）及底部 Tab 列（手機），讓消費者在任何子頁面都能快速切換功能。
**Entry points：** 網站 Header 右上角「我的帳戶」icon；登入後自動導向 `/account/orders`
**Primary user goal：** 快速導向目標功能頁面

#### Information Hierarchy

```
H1（最顯眼）：會員姓名 + 等級 Badge（若商家啟用等級系統）
H2（次要）：目前點數餘額快覽
Primary CTA：（依當前子頁面決定）
Supporting info：頭像圖示、左側導覽列各功能項目
```

#### Actual Copy

**會員資訊卡頭部**
- 點數快覽文字：`N 點`（點擊跳轉點數頁）
- 等級 Badge：顯示後台設定的等級名稱（如「金卡會員」）；無等級系統時不顯示
- 未上傳頭像：以姓名第一個中文字或英文首字母作為圓形頭像佔位

**左側導覽項目（依序）**
- `我的訂單`（預設進入頁）
- `個人資料`
- `收件地址`
- `點數 / 購物金`
- `我的優惠券`
- `心願清單`（後端啟用時顯示）
- `推薦好友`
- `帳號安全`
- `登出`（最底部，紅色文字）

**登出確認對話（EvomniAlert）**
- 標題：`確定要登出帳號嗎？`
- 說明：`登出後，購物車中的商品不會消失。`
- 確認按鈕：`確定登出`
- 取消按鈕：`取消`

**手機版底部 Tab 標籤**
- `訂單`
- `個人資料`
- `點數`
- `優惠券`
- `更多`

**「更多」展開後（Bottom Sheet）**
- `心願清單`（後端啟用時顯示）
- `收件地址`
- `推薦好友`
- `帳號安全`
- `登出`

**Session 過期 Toast（EvomniAlert）**
- 訊息：`登入已逾期，請重新登入。`

#### Component List

| 元件名稱 | 規格 / 變體 | 互動行為 |
|---------|-----------|---------|
| 會員頭像 | 圓形 40×40px；有圖片 / 首字母佔位兩種變體 | 點擊導向個人資料頁 |
| 左側導覽列 | 寬 220px；桌機完整版 / Tablet 60px icon-only 版 | 選中項顯示左側 3px 邊線強調；Hover 背景變深 |
| 底部 Tab 列 | 手機版（≤767px）固定底部，高度 56px，5 個 Tab | 點擊切換子頁面；「更多」展開 Bottom Sheet |
| 等級 Badge | 小型標籤，圓角；顏色對應等級設定 | 純展示，不可點擊 |
| Bottom Sheet | 從底部滑出；半透明遮罩覆蓋背景 | 點擊遮罩或拖動關閉 |

#### Screen States

| 狀態 | 觸發條件 | 呈現方式 |
|------|---------|---------|
| 已登入 - 一般 | 正常登入狀態 | 顯示頭像、姓名、點數；左側導覽或底部 Tab 可用 |
| 已登入 - 無等級系統 | 商家未設定等級 | 等級 Badge 不渲染 |
| 已登入 - 無點數功能 | 商家未啟用點數 | 點數快覽不渲染 |
| Session 過期 | 後端回傳 401 | Toast 提示 + 強制跳轉登入頁 |
| 未登入直接訪問 | 直接打開 `/account/*` 路由 | 301 跳轉 `/login?redirect=/account/orders` |

#### Interaction Annotations

| 使用者操作 | 結果 |
|-----------|------|
| 點擊左側導覽任一項目 | 切換對應子頁面，當前項目顯示選中樣式 |
| 點擊點數快覽 | 導向 Screen 6：點數 / 購物金 |
| 點擊「登出」 | 彈出 EvomniAlert 確認對話 |
| 確認登出 | 清除 Token → Toast 提示「已登出」→ 導向首頁 |
| Tablet 寬度 | 左側導覽自動收合為 60px icon-only；Hover 顯示文字 Tooltip |
| 手機版點擊「更多」| Bottom Sheet 從底部滑出 |

### Screen 2：我的訂單（列表）

**Purpose：** 讓消費者依狀態快速瀏覽所有訂單，並從卡片上直接執行常見操作（繼續付款、查看物流、申請退換貨）。
**Entry points：** 整體框架左側導覽「我的訂單」；登入後預設進入頁
**Primary user goal：** 找到目標訂單並了解其狀態或執行操作

#### Information Hierarchy

```
H1（最顯眼）：我的訂單
H2（次要）：狀態 Tab 列（含各 Tab 的待處理數量）
Primary CTA：（依訂單狀態：繼續付款 / 查看物流 / 申請退換貨）
Secondary CTA：查看訂單、繼續購物（Empty State）
Supporting info：訂單編號、下單日期、商品縮圖、金額、商品件數
```

#### Actual Copy

**Page Heading**
- 頁面標題：`我的訂單`

**狀態 Tab 標籤**
- `全部`
- `待付款`（有未付款訂單時顯示數字，如「待付款 2」）
- `待出貨`（有待處理訂單時顯示數字）
- `已出貨`（在途件數）
- `已完成`
- `已取消`
- `退換貨申請中`（有進行中申請時顯示數字）

**搜尋列**
- Placeholder：`搜尋訂單編號或商品名稱`

**訂單卡片欄位**
- 訂單編號前綴：`訂單編號：`
- 下單日期前綴：`下單日期：`
- 商品件數後綴：`共 N 件`
- 超過 3 項商品縮圖：`+N 件`

**操作按鈕（依訂單狀態）**
- 待付款：`繼續付款` / `取消訂單`
- 待出貨 / 備貨中：`查看訂單`
- 已出貨：`查看物流` / `確認收貨`
- 已完成：`查看訂單` / `申請退換貨` / `再次購買`
- 已取消：`查看訂單`
- 退換貨申請中：`查看申請進度`

**Empty State（各 Tab 無訂單）**
- 標題：`目前沒有[狀態名稱]訂單`（例：「目前沒有待付款訂單」）
- 說明：`逛逛商品吧，找到喜歡的就加入購物車！`
- 按鈕：`前往購物`

**搜尋無結果**
- 說明：`找不到符合「[關鍵字]」的訂單`
- 按鈕：`清除搜尋`

**確認收貨對話（EvomniAlert）**
- 標題：`確認已收到商品？`
- 說明：`確認後，訂單狀態將更新為完成。`
- 確認按鈕：`確認收貨`
- 取消按鈕：`取消`

**取消訂單對話（EvomniAlert）**
- 標題：`確定取消此訂單？`
- 說明：`訂單取消後無法恢復，若已付款將依退款政策處理。`
- 確認按鈕：`確定取消`
- 取消按鈕：`再想想`

**Toast 訊息**
- 確認收貨成功：`訂單已確認收貨`
- 取消訂單成功：`訂單已取消`

**Error States**

| 錯誤狀況 | 顯示給使用者的訊息 |
|---------|-----------------|
| 訂單列表 API 失敗 | `訂單資料載入失敗，請重新整理頁面。` |
| 確認收貨 API 失敗 | `操作失敗，請稍後再試。` |

#### Component List

| 元件名稱 | 規格 / 變體 | 互動行為 |
|---------|-----------|---------|
| 狀態 Tab 列 | 7 個 Tab；待處理項目顯示數字 Badge；Badge 色：待付款黃色、退換貨申請中紅色 | 點擊切換顯示對應訂單列表 |
| 搜尋輸入框 | 全寬；左側搜尋 icon | 輸入後即時篩選；清除按鈕（有輸入時出現）|
| 訂單卡片 | 含商品縮圖（最多 3 張）+ 金額 + 狀態 Tag + 操作按鈕 | 點擊卡片本體（非按鈕）導向訂單詳情 |
| 狀態 Tag | 各狀態對應色（待付款：橘色；已出貨：藍色；已完成：綠色；已取消：灰色） | 純展示 |
| 操作按鈕組 | 1–3 個按鈕；Primary 為主要操作；其他為次要按鈕 | 依狀態顯示對應按鈕 |
| Empty State | 插圖 + 說明文字 + CTA 按鈕 | 按鈕導向商品列表頁 |
| 分頁 / 載入更多 | 置底「載入更多」按鈕；每次載入 10 筆 | 點擊追加載入 |

#### Screen States

| 狀態 | 觸發條件 | 呈現方式 |
|------|---------|---------|
| 載入中 | 進入頁面、切換 Tab 時 | 訂單卡片區顯示 Skeleton 骨架 |
| 有訂單 | 正常狀態 | 卡片列表；Tab 顯示數量 Badge |
| Empty（特定 Tab）| 該狀態無任何訂單 | Empty State 插圖 + 說明 + 按鈕 |
| 搜尋無結果 | 搜尋詞無對應訂單 | 說明文字 + 清除搜尋按鈕 |
| API 錯誤 | 載入失敗 | 錯誤說明文字 + 重新整理按鈕 |

#### Interaction Annotations

| 使用者操作 | 結果 |
|-----------|------|
| 點擊訂單卡片（非按鈕區域）| 導向 Screen 3：訂單詳情 |
| 點擊「查看訂單」按鈕 | 導向 Screen 3：訂單詳情 |
| 點擊「繼續付款」 | 帶回付款資訊，導向金流頁面 |
| 點擊「查看物流」 | 新分頁開啟物流商追蹤頁；或展開內嵌物流時間軸 |
| 點擊「再次購買」 | 將訂單商品加入購物車；若有已下架商品，Toast 提示「部分商品已下架，無法加入購物車」|
| 點擊「申請退換貨」 | 導向退換貨申請表單（Part 3 負責） |

### Screen 3：訂單詳情

**Purpose：** 呈現單一訂單的完整資訊，包含狀態時間軸、收件人、付款、物流、商品明細、金額計算，並提供退換貨申請入口。
**Entry points：** 我的訂單列表點擊訂單卡片 / 操作按鈕
**Primary user goal：** 了解訂單目前狀態，以及取得物流追蹤資訊

#### Information Hierarchy

```
H1（最顯眼）：訂單編號 + 狀態 Tag
H2（次要）：訂單狀態時間軸
Primary CTA：查看物流 / 申請退換貨（依狀態顯示）
Secondary CTA：返回我的訂單、再次購買
Supporting info：收件人、付款方式、商品明細、金額計算
```

#### Actual Copy

**Page Heading**
- 返回連結：`返回我的訂單`
- 訂單編號標籤：`訂單編號：`

**狀態時間軸步驟文字**
- `訂單建立`
- `付款完成`
- `備貨中`
- `已出貨`
- `訂單完成`

**已取消訂單**
- 說明文字：`此訂單已於 YYYY/MM/DD 取消`
- 若有原因：`取消原因：[原因]`

**區塊標題**
- `收件資訊`
- `付款資訊`
- `物流資訊`
- `商品明細`
- `加購商品`（有加購商品時顯示）
- `金額計算`

**收件資訊欄位標籤**
- `收件人：`

## Assumptions

> 假設：會員中心 `/account/*` 所有子頁面均需登入，未登入時一律跳轉登入頁並帶 `redirect` 參數。

> 假設：心願清單功能預設關閉，需商家後台手動啟用。未啟用時前端完全不渲染導覽項目。

> 假設：等值購物金換算（點數頁）由後端 API 回傳 `points_exchange_rate` 值決定是否顯示；不顯示時不影響整體版面。

> 假設：推薦好友頁的點數獎勵數值（雙方各獲 N 點）由後台設定，前端直接顯示 API 回傳的數值。

> 假設：刪除帳號流程的 30 天冷靜期由後端 Cron Job 處理，前端只需送出申請並顯示說明。

> 假設：訂單詳情頁的物流狀態若後端有回傳節點資料，則展開內嵌時間軸；若僅有追蹤號碼而無節點資料，則顯示「查看物流」按鈕並新分頁開啟物流商追蹤頁。

> 假設：推薦記錄不顯示被推薦人的任何個人資訊（隱私保護）。

> 假設：「再次購買」功能：已下架商品靜默跳過，購買成功後 Toast 說明哪些商品因下架未能加入。
