## Part 1 — Design Brief

### Product Vision

訂單管理模組是 Evomni 電商後台的核心樞紐，負責訂單從建立到完成的完整生命週期，以及退換貨流程的全程追蹤。商家管理員透過此模組即時掌握訂單狀態、高效完成出貨揀貨、審核退換貨申請並觸發退款；消費者（包含未登入訪客）則可隨時查詢訂單進度、提交退換貨申請。整套系統自動串接金物流 API 與電子發票，讓商家不需離開後台即可完成稅務與退款作業。

### Target User

**Primary:** 商家管理員，每日處理數十至數百筆訂單，核心需求是快速找到特定訂單、批次出貨、審核退換貨申請，以及確保發票開立不遺漏。

**Secondary:** 消費者（含已登入會員與未登入訪客），需要在無需客服協助的情況下自助查詢訂單狀態或提交退換貨申請。

### Design Goals

| # | 目標 | 驗證方式 |
|---|---|---|
| 1 | 商家管理員能在 30 秒內找到任一筆訂單 | 請測試者找「三天前的某張 ATM 轉帳訂單」，計算從進入列表頁到開啟詳情頁所需時間 |
| 2 | 訪客無需登入，輸入訂單編號與手機末四碼即可查到訂單 | 請未登入使用者從首頁自行找到查詢入口並完成查詢，觀察是否需要提示 |
| 3 | 商家審核退換貨申請時，不需手算退款金額 | 觀察商家是否會拿計算機或另開 Excel，系統應直接顯示計算結果 |
| 4 | 「貨到付款未開立發票」的訂單一眼可辨 | 在含有多種付款方式的訂單列表中，請測試者找出所有未開立發票的訂單，計算誤判率 |
| 5 | 退換貨申請狀態全程可追蹤，商家與消費者不需打電話確認進度 | 在退換貨申請詳情頁，請測試者說出目前流程進行到哪一步、下一步由誰操作 |

### Design Principles

1. **狀態一眼可辨** — 訂單狀態、發票狀態、退換貨進度必須用顏色 + 文字雙重標示，不可只靠顏色區分。
2. **批次操作優先** — 高頻操作（列印出貨單、批次更新狀態、批次匯出）必須支援多選批次執行，減少重複點擊。
3. **計算透明** — 退款金額由系統自動計算並拆解呈現每個數字的來源，商家與消費者都能看懂計算依據。
4. **防呆大於提示** — 不可逆操作（取消已付款訂單、確認退款）必須有二次確認步驟，並說明後果。
5. **無縫通知** — 每個狀態變更後，系統自動觸發通知信，後台呈現最後一封通知的寄送時間，讓商家不需自行發信。

### Design System Reference

- **Figma Library:** Evomni Design System
- **Component framework:** 自訂元件庫（參考 Element Plus 規範）
- **Icon set:** Element Plus 內建圖示集
- **Font:** 系統字體堆疊（中文優先 PingFang TC / 微軟正黑體）

### Accessibility Requirements

- **Target WCAG level:** AA
- **Known constraints:**
  - 所有互動元素鍵盤可操作（Tab 順序明確）
  - 狀態 Tag 顏色搭配文字，色盲友善
  - 所有圖示按鈕附有 `aria-label` 或 Tooltip 文字說明

### Hard Constraints

- 不得在訪客訂單查詢結果頁顯示完整手機號碼（必須遮蔽中段）
- 不得在訪客訂單查詢結果頁顯示完整收件地址（僅顯示縣市）
- 取消已出貨或之後狀態的訂單按鈕必須呈現停用（`Disabled`）狀態，不可移除按鈕
- 退換貨退款金額計算結果為唯讀，後台管理員不可手動修改退款金額（特殊情況需另案處理）

### Out of Scope

- 超時未付款的逾時分鐘數設定介面（屬於一般商店設定模組）
- 會員中心的訂單歷史列表（屬於會員管理模組 Part 6）
- 金物流串接的帳號設定與 API 金鑰管理（屬於金物流串接設定模組）
- 電子發票的開立設定（發票類型、預設格式）

### Screen 1：訂單列表頁（後台）

**Purpose:** 商家管理員查看、搜尋、篩選所有訂單，並執行批次操作（列印出貨單、批次匯出）。
**Entry points:** 側邊欄「訂單管理」連結。
**Primary user goal:** 快速找到目標訂單並進入詳情，或批次選取多筆訂單執行出貨相關操作。

#### Information Hierarchy

```
H1 (most prominent): 訂單管理
Primary CTA: 新增訂單
Secondary CTA: 匯出訂單
Supporting info: 訂單筆數統計（各狀態 Tab 標籤旁的數量氣泡）
```

#### Actual Copy

**Page / Section Headings**
- 頁面標題: `訂單管理`
- Tab 列標籤（依序）: `全部` / `待付款` / `處理中` / `已出貨` / `配送中` / `已完成` / `已取消` / `退換貨申請`

**Button Labels**
- Primary action: `新增訂單`
- Secondary action: `匯出訂單`
- Destructive action: `取消訂單`（批次操作選單內，僅待付款可用）
- 批次操作觸發按鈕: `批次操作（已選 N 筆）`
- 批次操作選單項目: `列印出貨單` / `標記已出貨` / `匯出選取訂單` / `取消訂單`
- 篩選按鈕: `篩選條件`
- 重置篩選: `清除篩選`
- 套用篩選: `套用`

| 勾選全選後點擊「列印出貨單」 | 開啟新分頁顯示可列印的出貨單 PDF |

### Screen 2：訂單詳情頁（後台）

**Purpose:** 商家管理員查看單一訂單的完整資訊，更新訂單狀態，操作發票，以及列印出貨單。
**Entry points:** 訂單列表頁點擊任一訂單列；批次操作中點擊「查看詳情」。
**Primary user goal:** 確認訂單資訊正確後更新狀態（例如標記已出貨並填入物流單號）。

#### Information Hierarchy

```
H1 (most prominent): 訂單 #ORD-YYYYMMDD-XXXXXX  [狀態 Tag]
H2 (secondary): 產品明細區塊（左欄主要內容）
Primary CTA: 更新訂單狀態
Secondary CTA: 列印出貨單
Supporting info: 收件人資訊、付款資訊、物流資訊（右欄）
```

#### Actual Copy

**Page / Section Headings**
- 頁面標題: `訂單 #[訂單編號]`
- 區塊標題（左欄）: `產品明細` / `金額明細` / `退換貨記錄`
- 區塊標題（右欄）: `收件人資訊` / `付款資訊` / `物流資訊` / `備註`

**Button Labels**
- Primary action: `更新訂單狀態`
- Secondary actions: `列印出貨單` / `開立發票`（僅「未開立」或「開立失敗」顯示）
- Destructive action: `取消訂單`（「待付款」/「已付款」/「處理中」可用；「已出貨」後顯示但停用）
- 麵包屑: `訂單管理` > `訂單 #[訂單編號]`

**產品明細表格欄位**
- `產品圖` / `產品名稱與規格` / `單價` / `數量` / `小計`

**收件人資訊標籤**
- `收件人姓名` / `收件人手機` / `收件地址` / `備用聯絡電話`（選填）

**付款資訊標籤**
- `付款方式` / `付款時間` / `交易序號`（有值時顯示）

**物流資訊標籤**
- `配送方式` / `物流追蹤號` / `出貨時間`

**物流時間軸節點格式**
- `YYYY/MM/DD HH:mm — [狀態文字]（由 [操作人員] 操作）`
