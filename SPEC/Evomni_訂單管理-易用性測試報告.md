## 三、問題清單（依嚴重程度排序）

### 🔴 Critical（任務完全受阻）

---

**[C-01] 發票未開立警示邏輯錯誤 — 所有未付款訂單均顯示紅色警示**

- **影響：** Task T2（識別需補開發票訂單）
- **位置：** `components/MockData.jsx:14-15`
- **問題描述：** `INVOICE_STATUS.not_issued` 設定為 `color: 'danger'`、`icon: '⚠'`，導致訂單列表中所有「未開立」狀態的訂單（包含尚未付款的信用卡訂單）均以紅色 + 警示圖示顯示。商家無法區分真正需要補開發票的「貨到付款已完成」訂單，視覺雜訊嚴重影響稅務管理判斷。
- **根本原因：** 發票狀態顏色邏輯定義在全局靜態 `INVOICE_STATUS` 物件，無法依訂單付款方式和狀態組合動態調整。
- **建議修正：**

  `MockData.jsx` 中將 `not_issued` 的預設顏色改為中性色：
  ```js
  not_issued: { label: '未開立', color: 'info', icon: null },
  ```
  
  在 `Screen1.jsx` 表格渲染時動態覆寫：
  ```js
  const invStatusDisplay = order.invoice === 'not_issued' &&
    order.payment === '貨到付款' && order.status === 'completed'
      ? { label: '未開立', color: 'danger', icon: '⚠' }
      : INVOICE_STATUS[order.invoice] || {};
  ```

**[C-02] 前台退換貨申請表單（Screen 8）完全缺失**

- **影響：** 消費者在收到產品後無法自助提交退換貨申請
- **位置：** `html/訂單管理.html`（無 Screen 8 實作）；`components/Screen67.jsx` 中的 `Screen8CreateOrder` 是「後台手動建立訂單」，非前台申請表單
- **問題描述：** UX Spec Screen 8「退換貨申請表單（前台）」完全未實作。Screen 7 訪客查詢結果頁雖有「申請退換貨」按鈕（`Screen67.jsx:194`），點擊後執行 `onNavigate('guest-rma')`，但對應的路由和元件並不存在。
- **根本原因：** Prototype 僅實作後台 5 個畫面與前台 Screen 6/7，前台退換貨申請表單（Screen 8）尚未開發。
- **建議修正：** 依 UX Spec Screen 8 規格實作前台退換貨申請表單，包含申請類型選取、申請原因、佐證圖片上傳、ATM 退款帳戶欄位（條件顯示）。

### 🟠 Major（任務可到達但需要顯著猜測）

---

**[M-01] 訂單列表篩選面板缺少日期區間選項**

- **影響：** Task T1（尋找特定日期訂單）
- **位置：** `components/Screen1.jsx:151-176`
- **問題描述：** 篩選面板只提供「訂單狀態」「付款方式」「訂單來源」三個篩選維度，缺少 UX Spec 規定的「訂單日期區間」選項。商家無法快速定位某特定時段的訂單，必須手動掃描整個列表。
- **根本原因：** 篩選元件未實作日期選取器。
- **建議修正：** 在篩選面板加入日期區間選擇器（開始日期 + 結束日期）；預設顯示近 30 天；同時建議加入金額區間篩選（UX Spec 亦有定義）。

---

**[M-02] 點擊訂單列任意位置不會導向詳情頁**

- **影響：** Task T1（尋找並進入訂單詳情）— 需多次嘗試
- **位置：** `components/Screen1.jsx:239-276`
- **問題描述：** 訂單資料表格的 `<tr>` 元素只有 `onMouseEnter` / `onMouseLeave`（Hover 樣式），沒有 `onClick`。使用者點擊列中的「產品摘要」「訂單金額」「訂單狀態」等區域完全無反應，必須精確點擊訂單編號超連結或「查看詳情」按鈕才能進入詳情頁。
- **根本原因：** `<tr>` 未掛載 `onClick` 事件。
- **建議修正：**
  ```jsx
  <tr key={order.id}
    onClick={() => onNavigate('order-detail', { orderId: order.id })}
    style={{ cursor: 'pointer', ... }}
    ...>
  ```
  操作欄（勾選框、查看詳情、取消訂單）需加 `e.stopPropagation()` 防止冒泡。

---

**[M-03] 退換貨申請列表缺少狀態 Tab Bar，改用篩選組合**

- **影響：** Task T4（找到待審核的退換貨申請）— 初次進入需摸索篩選操作
- **位置：** `components/Screen3.jsx:55-87`
- **問題描述：** UX Spec 要求退換貨申請列表使用 8 個狀態 Tab（全部 / 待審核 / 待消費者寄回…），每個 Tab 附數量氣泡，讓商家一眼看出各狀態申請數量。Prototype 改用「申請類型 Radio + 狀態 Select 下拉」的組合篩選，沒有數量氣泡，無法一眼掌握各狀態申請量。
- **根本原因：** Screen 3 以通用篩選元件取代 Tab Bar 設計。
- **建議修正：** 參照 Screen 1 的 Tab Bar 實作方式，在 Screen 3 加入 8 個狀態 Tab，各 Tab 標籤旁顯示申請數量氣泡。

---

**[M-04] 退換貨申請詳情頁缺少流程進度條**

- **影響：** Task T7（判斷目前流程位置）— 商家需閱讀狀態 Tag 和操作按鈕組合才能判斷
- **位置：** `components/Screen4.jsx:180-188`
- **問題描述：** UX Spec 規定頁面頂部必須有水平流程步驟條（退貨 7 步驟 / 換貨 7 步驟），讓商家一眼看出目前流程在哪個階段、下一步應由誰操作。Prototype 只顯示頁面標題 + 狀態 Tag，商家需理解所有狀態鍵值才能判斷流程位置。
- **根本原因：** Step Progress Bar 元件未實作。
- **建議修正：** 在 Screen 4 頁面標題下方加入水平進度條，依申請類型（退貨 / 換貨）顯示對應步驟，已完成節點打勾，當前節點醒目標示。
