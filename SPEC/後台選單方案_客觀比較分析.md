# 後台選單架構方案 — 客觀比較分析

**文件性質：** 中立評估，不預設結論
**版本：** v1.0 | **日期：** 2026/06/10
**背景：** 先前分析文件因方案 A 已實作，存在確認偏誤風險，本文重新以對等標準評估兩方案

---

## 前言：評估標準說明

本文對兩方案採用相同的評估嚴格度：

- 方案 A 的「角色權限未來可解決選單過長」與方案 B 的「URL 路由工程可解決情境持久性」，同樣視為「尚未完成的工程投資」，不主動幫任一方案預支未兌現的優點
- 兩方案的缺點都用「能否被設計解法緩解」來評等，而非直接判定為致命問題
- 共同面臨的問題（與選單架構無關）單獨列出，不歸咎於任一方案

---

## 一、方案 A — 分區整合式選單

### 核心前提

所有功能列在同一份選單中，電商與形象站功能以區段小標（section label）做視覺分組。選單是靜態的，不隨使用者的動作改變結構。

### 真實優勢

**1. 跨功能導覽成本恆為一次點擊**
不論使用者目前在訂單管理還是品牌形象站，切換到任何其他功能都是直接點擊，沒有前置步驟。對於同一個工作流程需要橫跨多個功能的使用者，這是結構性的效率優勢。

**2. 不存在「情境狀態」這個工程問題**
選單不儲存任何狀態，URL 路由單純，多分頁行為一致，重新整理後頁面如常，書籤永遠有效。這些是由架構保證的特性，不需要額外工程投資。

**3. 通知徽章全程可見**
訂單待處理數、訊息通知等徽章，不論使用者在哪個功能頁面都能在側邊欄看到。使用者不需要因為在 CMS 工作就失去對電商狀態的感知。

**4. 對混合角色使用者心智負擔低**
對同時管理電商與網站的店長或老闆，選單的樣子與他們對業務的認知一致：這是一套系統，不是兩個系統。不需要在腦中維持「我目前在哪個空間」的狀態。

**5. 已完整實作，遷移成本為零**
在資源有限的情況下，現有實作的穩定性是真實的優勢，不只是惰性。

### 真實劣勢

**1. 選單長度沒有天然上限**
目前約 15 個一級項目（含區段標）。電商功能若再新增行銷工具、會員分級、金流設定等，選單可能成長至 20 項以上。方案 A 對功能增長沒有結構性的容納機制，唯一的手段是增加子層級，但這同樣有深度限制。

**2. 區段小標是弱組織信號**
區段小標是靜態文字，不可點擊，視覺重量低，使用者在快速掃描時容易忽略。它提供的組織感比真正的分層導覽弱，當選單項目增多時，這個弱點會被放大。

**3. 共用功能的歸屬不直覺**
產品、文章、顧客放在所有區段標籤之前的「無標籤區」，對新使用者而言，這個位置沒有明確的語義——它們不屬於電商系統，不屬於網站設計，也不屬於設定，但它們確實存在。對理解系統架構有興趣的使用者，這個設計難以解釋。

**4. 對單一功能角色產生持續視覺噪音**
倉管每天看到媒體庫、品牌形象站等完全無關的功能。這個問題依賴「未來角色權限系統」才能真正解決，在此之前是一個已知的使用體驗缺陷。

**5. 沒有域邊界提示，跨域誤操作沒有防護**
CMS 編輯在方案 A 下，如果帳號權限設定不完整，可能看到並誤改電商設定（售價、庫存）。方案 A 不提供任何「你正在跨越域邊界」的提示機制。

## 二、方案 B — 情境切換式選單

### 核心前提

選單頂部有情境切換器，使用者在「電商後台」與「形象站」之間切換，選單根據情境只顯示對應功能。兩個情境共用五個項目（儀表板、產品、文章、顧客、全域設定）。

### 真實優勢

**1. 更短的選單降低每次決策的認知負荷**
認知心理學研究（Hick's Law）顯示，選項數量增加時決策時間呈對數成長。方案 B 的每個情境約 9 項 vs 方案 A 的 15 項，是可量化的認知效益，對所有使用者都有正面影響，不只對單一功能角色。

**2. 明確的域邊界提供架構透明度**
電商後台與形象站的邊界在系統底層是真實存在的——產品資料表、訂單資料表、CMS 頁面各自屬於不同的業務域。方案 B 把這個架構現實呈現給使用者，使用者對「為什麼這個功能在這裡」的疑問，方案 B 提供了更一致的答案。

**3. 自然對應角色分工，為未來擴展預留空間**
當商家規模成長，員工開始有明確分工時，方案 B 的情境模型直接對應「這個員工管電商、那個員工管網站」的組織結構。角色權限系統的實作在方案 B 的架構下更自然——情境即是第一層的分類，角色權限在此之上再做細化。

**4. 抑制跨域誤操作**
情境切換器作為一個有意識的動作，在使用者跨越域邊界時提供了一個心理檢查點。這在未來若有欄位層級的域控制時尤為重要，且即使在全欄位開放的前提下，至少讓使用者意識到自己正在跨域作業。

**5. 功能成長的長期容錯能力更強**
若電商端新增 5 個功能、CMS 端新增 3 個功能，方案 B 的兩個情境選單各自成長，使用者在各自的情境中看到的選單長度仍然可控。方案 A 的單一清單在同樣的增長下會更快到達可用性臨界點。

**6. 選單的視覺聚焦感有助於任務完成感**
使用者在 EC 後台時，看到的都是與電商相關的功能，視覺上感受到「現在就是在處理電商的事」，這種聚焦感對效率有心理層面的正面影響，特別是對單一功能角色。

### 真實劣勢

**1. 跨域任務強制插入中繼步驟**
任何需要橫跨電商與形象站功能的工作流，都必須包含至少一次情境切換。對 Evomni 商家而言，這類跨域任務（上架後更新網站選單、看數據後調整首頁展示、整理圖片後更新商品）是高頻任務，這個成本會持續積累。

**2. 情境狀態需要工程投資才能正確運作**
多分頁行為、重新整理後的情境持久性、跨裝置一致性、書籤有效性，這些問題在方案 B 下都需要把情境狀態編碼至 URL 或搭配可靠的 session 儲存機制。這是方案 A 不需要的工程成本，且若處理不好，使用者體驗的損害是隱形且難以察覺的。

**3. 共用項目出現在兩個情境，削弱情境切換的設計意義**
方案 B 聲稱「切換情境後只看到當前情境的功能」，但產品、文章、顧客等五個共用項目同時出現在兩個情境。這讓兩個選單的差異比預期小，且使用者容易對「為什麼這些在兩邊都有」感到困惑。共用項目的存在不斷提醒使用者「這兩個情境的邊界其實是模糊的」。

**4. 誤觸情境切換後的可復原性差**
情境切換後的視覺變化（選單重組）對不熟悉此機制的使用者等同於系統異常，且還原路徑不直覺（需要找到切換器再切回來）。這個問題在平板觸控操作下因誤觸機率提高而被放大。

**5. 行動裝置的切換器設計尚無解法**
在漢堡選單模式（< 1024px）下，情境切換器若放在覆蓋層內，切換動作需要多步操作；若放在 header，佔用有限空間。目前方案 B 的設計尚未解決這個問題，而且沒有一個令人滿意的標準解法。

**6. 通知徽章的跨域可見性存在缺口**
在形象站情境工作時，EC 後台的訂單通知徽章不可見。對同時管理兩域的使用者，這是一個可能造成訂單延誤的功能性問題，不只是體驗問題。

## 三、兩方案共同面臨的問題

以下問題與選單架構選擇無關，無論採用哪個方案都需要獨立解決：

**子選單觸控行為設計**
訂單管理、行銷中心等有子項目的父項，在觸控裝置上「點擊是展開還是導向」的歧義問題，是選單互動設計問題，不是架構問題。

**媒體庫的歸屬問題**
媒體庫在邏輯上屬於形象站，但商品管理也需要用到它。無論方案 A 或 B，這個功能的歸屬都需要設計決策——方案 B 的差別只是讓這個問題更顯著。

**首次使用引導**
新商家登入後台的 onboarding 體驗，與選單架構有關聯，但主要仍是獨立的設計工作。

## 四、兩方案的適用情境

這是核心判斷框架，不同的條件下兩個方案各有更合理的選擇：

### 方案 A 更合適的條件

- 商家規模小（1–3 人），員工角色高度重疊，同一個人處理電商與 CMS
- 工作流程高頻跨域（上架一定要更新網站選單、看數據一定要調整網站展示）
- 開發資源有限，不希望引入全域情境狀態管理的工程複雜度
- 產品當前階段以快速迭代為主，不希望為架構決策付出遷移成本
- 商家對後台的認知是「一套系統」，不區分電商與 CMS

### 方案 B 更合適的條件

- 商家規模較大（5 人以上），員工有明確的角色分工
- 電商端與 CMS 端的功能各自持續增長，選單長度已成問題
- 有系統性的角色權限管理需求，情境切換可作為第一層的存取控制
- 商家自身就以「電商後台」與「官網後台」作為兩個獨立的心智空間
- 開發資源足夠投入情境狀態管理的工程

## 五、誠實的結論

**方案 A 和方案 B 解決的是同一個問題的不同版本。**

方案 A 的核心假設是：使用者不需要知道 EC 和 CMS 的邊界，他們需要的是快速找到功能。這個假設對現階段的目標客群（SMB、小團隊、混合角色）是合理的。

方案 B 的核心假設是：使用者應該知道 EC 和 CMS 是兩個域，明確的邊界能幫助他們更好地理解系統並減少錯誤。這個假設對成長中的商家和有角色分工的團隊是合理的。

**兩個假設都不是錯的，只是服務的使用者樣貌不同。**

| 決策維度 | 方案 A 有利 | 方案 B 有利 |
|---------|-----------|-----------|
| 現階段目標客群（SMB、1–3 人）| 是 | 否 |
| 長期功能成長的選單可維護性 | 否 | 是 |
| 跨域工作流的效率 | 是 | 否 |
| 角色分工明確的組織 | 否 | 是 |
| 目前工程成本 | 低 | 中高 |
| 行動裝置可用性 | 良好 | 尚待解決 |
| 通知可見性 | 完整 | 存在缺口 |
| 認知負荷（每個情境的選單長度）| 較高 | 較低 |
| 誤觸後的可復原性 | 高 | 低 |
| 架構透明度 | 低 | 高 |

**若以 Evomni 目前的產品階段（SMB 優先、快速迭代、工程資源有限）作為決策依據，方案 A 在現階段的風險更低。**

**但這個結論的有效期限是有條件的：**
若 Evomni 在 12–18 個月後開始服務有明確員工分工的中型商家，或者電商端與 CMS 端各自的功能項目再增加 30% 以上，方案 B 的架構前提將更加成立，屆時的遷移成本也會隨功能數量增加而提高。若存在方案 B 是長期方向的共識，現在就做不是錯誤，只是需要同步投入對應的工程成本。
