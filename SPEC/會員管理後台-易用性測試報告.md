## 三、問題清單（依嚴重程度排序）

### 3.1 Major 問題

**[M1] 等級設定 Dialog 缺少三個必要欄位**

- 影響任務：T6
- 位置：`html/會員管理後台.html`（MemberLevelPage 的 Dialog 定義，約 800–900 行）
- 問題描述：UX Spec Screen 3 定義的「新增/編輯等級」Dialog 必須包含五個欄位：等級名稱、升等門檻（累計消費金額）、等級有效期（永久或自訂月數）、保級條件（保留等級所需年度消費額）、入會禮/升等禮（可選填）。目前 Dialog 只實作了「等級名稱」與升等條件（金額）兩個欄位。使用者嘗試建立完整的等級規則時，無法設定有效期和保級機制，導致任務無法完整完成。
- 根本原因：Dialog 實作未對照 UX Spec 全部欄位清單，屬於規格漏接。
- 建議修正：補齊欄位（等級有效期：永久 / 自訂 N 個月；保級條件：金額輸入 + 計算期間；入會/升等禮：文字輸入或選擇已建立的優惠券）。

---

**[M2] 等級設定列表缺少拖曳排序功能**

- 影響任務：T6（發現問題時的探索行為）
- 位置：`html/會員管理後台.html`（MemberLevelPage 的等級 table，約 750–800 行）
- 問題描述：UX Spec Screen 3 明確說明「等級顯示順序可透過拖曳排序（drag to reorder）調整」。目前 Prototype 的等級列表中沒有任何拖曳 handle（通常為垂直六點 icon 或 = 符號），也沒有任何說明文字提示可以排序。非技術用戶不會猜到此功能。
- 根本原因：Prototype 未實作拖曳排序功能。
- 建議修正：在每列左側加入拖曳 handle icon；若 Prototype 階段不實作實際拖曳，至少加入視覺 handle 和 tooltip「拖曳以調整等級順序」。

---

**[M3] Screen 01 欄位標題「註冊日期」與規格「加入日期」不符**

- 影響任務：T1
- 位置：`html/會員管理後台.html`（CustomerListPage 的 table header，約 350–420 行）
- 問題描述：UX Spec Screen 1 及 PRD 一致定義此欄位名稱為「加入日期」，代表顧客在系統中留下第一筆記錄的日期（首次消費或手動建立）。Prototype 使用「註冊日期」，語意偏向主動填表的帳號建立日，對未在系統中正式「註冊」的顧客（如以訪客身分下單的顧客）會造成困惑。
- 根本原因：Prototype 撰寫時沿用了技術性詞彙，未對照 UX Spec 文案。
- 建議修正：將欄位標題從「註冊日期」改為「加入日期」；同步確認 CustomerSettingsPage（Screen 10）的欄位設定面板也使用相同名稱。

---

### 3.2 Minor 問題

**[m1] Screen 01 匯出 Toast 訊息包含 Emoji**

- 影響任務：T1（次要觀察）
- 位置：`html/會員管理後台.html`（CustomerListPage 的匯出按鈕 handler，約 500 行附近）
- 問題描述：點擊「匯出全部」或「匯出選取」後觸發的 Toast 訊息包含 Emoji 字元，違反 CLAUDE.md 最高原則（所有輸出一律不得出現 Emoji）。
- 根本原因：Toast 訊息撰寫時未移除 Emoji。
- 建議修正：移除 Toast 內所有 Emoji；改以純文字表達，例如「已排程匯出，檔案就緒後將自動下載」。

---

**[m2] Screen 09 生日賀卡預設模板主旨包含 Emoji**

- 影響任務：T7（編輯模板時可觀察到）
- 位置：`html/會員管理後台.html`（DEFAULT_TEMPLATES.birthday.subject，約 1679 行）
- 問題描述：生日賀卡的預設主旨為「生日快樂！{{member_name}} [生日蛋糕 Emoji]」，Emoji 字元違反 CLAUDE.md 最高原則。此主旨在模板預覽中可見，使用者可能直接採用而不修改，導致實際發信時包含 Emoji。
- 根本原因：預設模板撰寫時未遵循無 Emoji 規則。
- 建議修正：主旨改為「生日快樂！{{member_name}}，本月消費享雙倍點數回饋」或類似純文字表達。

---

**[m3] Screens 03–10 的 H1 顏色與 Screen 01 不一致**

- 影響任務：視覺一致性，不影響功能
- 位置：`html/會員管理後台.html`（各 Screen 的 H1 樣式，如 TagsPage 約 1097 行、MemberSettingsPage 約 1511 行 等）
- 問題描述：Screen 01（顧客管理）的頁面 H1 使用 `T.textPrimary`（深灰 #303133），而 Screens 03–10 的 H1 均使用 `T.blue`（#409EFF）。跨畫面的視覺語言不統一，形成非預期的對比，可能讓使用者誤以為不同顏色代表不同層級或性質的頁面。其他 Prototype（如廣告投放工具）的 H1 統一使用 `T.textPrimary`。
- 根本原因：Screens 03–10 撰寫時未對照 Screen 01 的既有樣式。
- 建議修正：將 Screens 03–10 的 H1 顏色統一改為 `T.textPrimary`。

---

### 3.3 Enhancement

**[e1] 分眾篩選的功能鎖定邊界不清晰**

- 影響任務：T5（探索時路過分眾篩選頁）
- 位置：`html/會員管理後台.html`（SegmentPage，約 1260–1325 行）
- 問題描述：頁面頂部有 UpgradeLockBanner，標示「進階會員分眾系統」為進階方案功能。但頁面上的「預覽符合人數」與「發送優惠券」按鈕在用戶加入篩選條件後仍為可用狀態（僅在無條件時 disabled）。使用者會困惑：這個功能到底是鎖定的還是可以用的？
- 建議修正：確認產品定義範圍——若整頁均鎖定，按鈕應全部 disabled 並顯示 upgrade tooltip；若僅「發送優惠券」屬進階，則 Banner 文案應更精確說明鎖定的具體功能。

---

**[e2] 顧客詳情頁「調整點數」入口發現困難**

- 影響任務：T3
- 位置：`html/會員管理後台.html`（CustomerDetailPage 的點數卡片右上角，約 600 行附近）
- 問題描述：管理員若需要手動調整點數，需要進入顧客詳情頁，找到「點數」資訊卡，才能看到右上角的「調整點數」按鈕。對初次使用的管理員而言，「調整點數」是一個隱藏在三欄式資訊卡群組中的次要 CTA，不容易第一時間發現。
- 建議修正：在詳情頁頂部的操作按鈕區（加入黑名單等按鈕旁）增加「調整點數」快速入口，或在點數卡片標題旁加入更明顯的按鈕。

## 五、受測者語錄

以下為 AI 根據實際 UI 文字與互動流程，模擬非後台使用者的可能反應：

1.「我進來第一眼就看到一個表格，上面有很多名字和 Email，還不錯，我大概知道這是在管理客戶的地方。」（Task 1，顧客列表印象良好）

2.「欸，這個『註冊日期』是什麼日期？是他在系統裡自己填表的日期嗎？還是第一次買東西的日期？我搞不太清楚。」（Task 1，欄位標題語意不明）

3.「我想幫這位客人手動加點數，但我在這個頁面上找了很久，一直找不到在哪裡操作。後來才發現原來要點進他的個人頁面，再找到點數那個區塊裡面的一個小按鈕。」（Task 3，調整點數入口隱蔽）

4.「黑名單那個功能我覺得設計得不錯，它要我寫原因才能加進去，這樣比較安全，不會誤按。」（Task 4，黑名單確認流程評價正面）

5.「我要改驗證方式，但不確定要去哪裡找。我試著點了幾個地方，最後在『設定』裡面的『會員設定』找到了，但我花了有點時間。」（Task 5，設定 Hub 導覽路徑需探索）

6.「我想設定新的會員等級，但裡面能填的欄位好像不太夠，感覺少了什麼。我想設定這個等級要維持多久、要怎麼樣才能保住等級，但找不到地方填。」（Task 6，等級 Dialog 缺欄位）

7.「這個改信件模板的功能我覺得很好用，右邊有預覽，我改完就能馬上看到效果。那個變數的提示也很清楚。」（Task 7，通知信模板編輯評價正面）

8.「我去分眾篩選那個頁面，上面有一個橫幅說這是進階方案的功能，但底下我還是可以按那些按鈕，讓我有點搞不懂這個功能到底能不能用。」（Task 5 探索期，分眾篩選功能邊界困惑）
