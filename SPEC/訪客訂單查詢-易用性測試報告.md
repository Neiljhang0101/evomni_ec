## 三、問題清單（依嚴重程度排序）

---

### 🔴 Critical — 按鈕點擊後進入未定義 State，頁面主體消失

- **影響任務：** T2（核心查詢流程完全阻斷）
- **位置：** `html/訪客訂單查詢.html:252`
- **問題描述：** QueryForm 的「查詢訂單」按鈕 `onClick` 觸發 `onStateChange('loading-demo')`，但 `'loading-demo'` 不在 STATES 陣列中，且不符合 `showForm`、`showEmpty`、`showResult` 任一條件判斷。導致頁面進入此 State 後，所有主體內容（表單、空態、結果）全部不顯示，使用者面對空白頁面。
- **根本原因：** STATES 陣列缺少 `'loading'` 或 `'loading-demo'` 項目；按鈕 onClick 設計的 loading 中繼態從未被加入可識別的 State 列表。
- **建議修正：**
  ```jsx
  // 方案A（最快）：按鈕直接導向示範結果 State
  onClick={() => onStateChange('delivering')}

  // 方案B（推薦）：加入 loading State，延遲後切換到結果
  onClick={() => {
    onStateChange('loading');
    setTimeout(() => onStateChange('delivering'), 1200);
  }}
  // 同時在 STATES 加入 { key: 'loading', label: '查詢中...' }
  // 並在 showForm 條件加入 || state === 'loading'
  ```

### 🔴 Critical — 表單未實作空欄位前端驗證

- **影響任務：** T2
- **位置：** `html/訪客訂單查詢.html:243-265`
- **問題描述：** 按下「查詢訂單」時，不論欄位是否填寫，都直接呼叫 `onStateChange('loading-demo')`。Spec 明訂：「點擊查詢訂單（有空白欄位）→ 欄位下方顯示對應錯誤提示，不送出 API」。非專業使用者若不小心未填任何欄位即點擊，得到空頁面，完全不知道問題所在。
- **根本原因：** Prototype 未模擬欄位驗證邏輯，按鈕直接觸發 State 切換。
- **建議修正：** 加入欄位空值判斷，空值時顯示欄位下方錯誤提示，不切換 State。

---

### 🟠 Major — fail 狀態顯示欄位級錯誤，違反防枚舉攻擊安全原則

- **影響任務：** T3
- **位置：** `html/訪客訂單查詢.html:219-221`
- **問題描述：** `fail` 狀態下，訂單編號欄位下方顯示「請輸入有效的訂單編號」。這直接透露「訂單編號欄位有問題」，違反 Spec Hard Constraints：「驗證失敗提示不透露是哪個欄位輸入錯誤（防枚舉攻擊）」。全頁 Alert 已正確顯示統一錯誤文案，欄位級錯誤屬多餘且有安全風險。
- **根本原因：** `fail` 狀態複用了欄位驗證失敗的錯誤提示邏輯，兩種錯誤情境未區分。
- **建議修正：** 移除 `fail` 狀態時訂單編號欄位下方的 `<p>` 錯誤提示（line 219-221）。全頁 Alert 已足夠。

---

### 🟠 Major — 頁面有兩個 H1 標籤，Screen Reader 會讀取兩次頁面標題

- **影響任務：** T2、T3（觸及表單態時）
- **位置：** `html/訪客訂單查詢.html:189`（QueryForm 內）與 `html/訪客訂單查詢.html:497`（頁面標題帶）
- **問題描述：** 當表單顯示時，頁面同時存在兩個 `<h1>`：
  - 頁面標題帶（line 497）：`查詢您的訂單`
  - QueryForm 卡片內（line 189）：`查詢您的訂單`
  兩者文字相同，視覺上看似正常，但 Screen Reader 會讀兩次主標題，違反 WCAG AA。
- **根本原因：** QueryForm 內部原本為獨立使用設計，移入頁面結構後未移除或降級為 H2。
- **建議修正：** 將 QueryForm 內的 H1（line 189）改為 `<p>` 或移除；頁面只保留最外層的 H1。

---

### 🟡 Minor — 結果卡片缺少 border-radius: 12px

- **影響任務：** T4、T5
- **位置：** `html/訪客訂單查詢.html:334`
- **問題描述：** Spec 明訂結果容器「border-radius 12px」。Prototype 的結果卡片 `<div>` 無 `borderRadius` 屬性，呈現直角。
- **建議修正：** 在 line 334 的 div style 加入 `borderRadius: 12` 或 `12px`。

---

### 🟡 Minor — 「申請退換貨」按鈕樣式與 Spec 不符

- **影響任務：** T5
- **位置：** `html/訪客訂單查詢.html:424`
- **問題描述：** Spec 規定按鈕使用「Secondary 樣式，`var(--danger-100)` 背景」，視覺上應帶有輕微紅色，傳達謹慎操作的語意。Prototype 使用 `background: '#fff'`（白底），外觀與一般次要按鈕無異，使用者較難直覺判斷這是一個需要注意的操作。
- **建議修正：** 將 background 改為 `var(--danger-soft)` 或 `#fef2f2`，border 改為 `1px solid var(--danger)`，color 改為 `var(--danger)`。

---

### 🟡 Minor — QueryEmpty 標題與頁面 H1 重複，標題結構冗餘

- **影響任務：** T6
- **位置：** `html/訪客訂單查詢.html:284`（QueryEmpty H2）與 `html/訪客訂單查詢.html:498`（頁面 H1）
- **問題描述：** `no-result` 狀態下，頁面 H1 顯示「查無符合的訂單」，QueryEmpty 卡片內的 H2 也是「查無符合的訂單」，兩者重複出現，視覺層次冗餘。
- **建議修正：** 二選一：（A）移除 QueryEmpty 內的 H2，卡片改以說明文字為主；或（B）QueryEmpty 內的 H2 改為更具體的次標題，如「請確認查詢資訊」。

## 五、受測者語錄

以下為根據實際 UI 流程，模擬非專業消費者操作時**可能說出**的話語：

5. 「這個『申請退換貨』……看起來跟其他按鈕一樣，應該可以按吧？」（T5，按鈕視覺缺乏 Danger 語意）

9. 「我的訂單取消了，這裡說付款逾時自動取消，庫存回補了，那我是要重新訂一次嗎？沒有重新訂購的連結嗎？」（取消態，缺少後續行動引導）
