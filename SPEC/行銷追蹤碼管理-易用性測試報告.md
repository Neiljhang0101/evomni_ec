## 三、問題清單（依嚴重程度排序）

### P-01 — 停用 Meta Pixel 時，若 CAPI 已開啟但尚未儲存，確認框不顯示 CAPI 同步停用警示

**嚴重程度：** 🟠 Major
**影響任務：** T3（Conversion API 設定）、T6（停用工具）
**位置：** `html/行銷追蹤碼管理.html:718`

```jsx
// 目前條件只看 capi.saved，忽略 capi.on=true 但尚未儲存的情況
{confirmDialog.tool === 'pixel' && tools.capi.saved && (
  <p style={{ marginTop: 10, color: '#E6A23C', fontSize: 13, lineHeight: 1.7 }}>
    注意：停用 Meta Pixel 後，Conversion API（FB 伺服器轉換 API）也將同步停用。
  </p>
)}
```

**問題描述：** 若商家已將 CAPI 開關切至 ON 但尚未點「儲存此設定」，此時停用 Meta Pixel 的確認框不會顯示「CAPI 也將同步停用」的警示。但在 `handleConfirmDisable` 中（第 430-435 行），停用 Pixel 的邏輯會同時將 `capi.on` 設回 false。結果是 CAPI 在用戶不知情的情況下被靜默關閉，違反 UX Spec 的提示要求。

**根本原因：** 顯示條件只判斷 `capi.saved`，未考慮 `capi.on === true` 的中間狀態。

**建議修正：** 將條件改為 `tools.capi.on || tools.capi.saved`。

### P-02 — CAPI 卡片名稱在 Prototype 中不一致

**嚴重程度：** 🟠 Major
**影響任務：** T3（Conversion API 設定）
**位置：** `html/行銷追蹤碼管理.html:384, 592, 436`

| 出現位置 | 名稱 |
|---|---|
| TOOL_NAMES（用於確認框標題、Toast）行 384 | `Conversion API（FB 伺服器轉換 API）` |
| ToolCard title 行 592 | `Meta Conversion API（FB 伺服器轉換 API）` |
| Pixel 停用 Toast 行 436 | `Meta Pixel 及 Conversion API（FB 伺服器轉換 API）已停用` |
| UX Spec（Screen 4 標題）| `Meta Conversion API（FB 轉換 API）` |
| UX Spec（Screen 7 確認框標題）| `確認停用 Conversion API？` |

**問題描述：** Prototype 全文使用「FB 伺服器轉換 API」，而 UX Spec 使用「FB 轉換 API」。雖然前者更技術精確（符合 CLAUDE.md 白話化原則），但 Prototype 內部各出現位置之間本身也不統一（有時有 `Meta` 前綴，有時無）。商家在操作過程中可能無法判斷卡片標題、確認框標題、Toast 訊息是否在描述同一個功能。

**根本原因：** 命名規則未在 Prototype 中統一定義，多處獨立撰寫。

**建議修正：** 統一使用 TOOL_NAMES 作為唯一的名稱來源，所有顯示文字都從此 object 取值。建議名稱定案為 `Meta Conversion API（FB 轉換 API）`，與 UX Spec 一致，並更新 CLAUDE.md 的白話化規則以確認「FB 轉換 API」為核准用語。

### P-03 — 儲存按鈕在欄位空白時未 Disabled

**嚴重程度：** 🟡 Minor
**影響任務：** T1（GA4 設定）、T2（Pixel 設定）、T4（GTM 設定）
**位置：** `html/行銷追蹤碼管理.html:203-218`

**問題描述：** UX Spec Screen 2 規定「展開未填（初次）| 儲存按鈕 Disabled」，但 Prototype 中的 `SaveBtn` 元件在欄位空白時仍可點擊。點擊後會顯示正確的錯誤訊息，任務仍可完成，但空欄位時可點擊的按鈕會讓用戶多走一步（「點了才知道要填什麼」），不如欄位有內容後按鈕才亮起來直觀。

**根本原因：** `SaveBtn` 無 disabled prop；父層也未根據欄位值傳入 disabled 狀態。

**建議修正：** 為 `SaveBtn` 加入 `disabled` prop；在各工具的 Card 中根據 ID/Token 欄位是否有值來控制按鈕的 disabled 狀態。

**嚴重程度：** 🟡 Minor
**影響任務：** T5（查看事件統計）
**位置：** `html/行銷追蹤碼管理.html:255`

**問題描述：** 表格欄位標題目前為「對應 FB 像素（Pixel）事件」，UX Spec 規定為「對應 FB Pixel 事件」。原型的寫法符合 CLAUDE.md 白話化原則（英文術語配中文說明），但與規格文件不一致。建議對齊規格後更新 CLAUDE.md 核准用語清單。

**根本原因：** 白話化原則應用無問題，但 UX Spec 尚未同步更新。

**建議修正：** PM 確認核准用語後，統一更新 Prototype 和 UX Spec；或更新 Spec 為「對應 FB 像素（Pixel）事件」。

### P-05 — 停用確認框標題對 CAPI 過於冗長

**嚴重程度：** 🟡 Minor
**影響任務：** T3（停用 CAPI）
**位置：** `html/行銷追蹤碼管理.html:708`

**問題描述：** 停用 CAPI 時確認框標題為「確認停用 Conversion API（FB 伺服器轉換 API）？」，UX Spec 規定為「確認停用 Conversion API？」。括號內的補充說明使標題超過一行，在 Dialog header 的有限空間中顯示過擁擠。

**根本原因：** TOOL_NAMES 中 CAPI 的完整名稱被直接插入 Dialog 標題中。

**建議修正：** 建立獨立的 `TOOL_DIALOG_NAMES` 對應，CAPI 的 Dialog 標題使用較短的 `Conversion API`。

### P-06 — 幣別選項 Dash 格式（"-" vs "—"）

**嚴重程度：** 🟡 Minor
**影響任務：** T2（Pixel 設定）
**位置：** `html/行銷追蹤碼管理.html:389-392`

**問題描述：** Prototype 幣別選項為 `TWD - 新台幣`（連字符）；UX Spec 規定為 `TWD — 新台幣`（全形破折號 em-dash）。視覺上連字符顯示較為擁擠，Em-dash 更符合印刷排版慣例。

**根本原因：** 複製時使用了半形連字符而非 em-dash。

**建議修正：** 將三個幣別選項的 `-` 改為 `—`（U+2014）。

---

### P-07 — 事件統計說明文字顏色偏淺

**嚴重程度：** 🟡 Minor
**影響任務：** T5（查看事件統計）
**位置：** `html/行銷追蹤碼管理.html:693`

**問題描述：** 說明文字使用 `color: '#909399'`（placeholder 色），而 UX Spec Design System Token Reference 規定次要說明文字應使用 `#606266`（Text secondary）。`#909399` 為 placeholder/disabled 色，用於一般說明文字時對比度略不足（WCAG AA 要求 4.5:1 比率，`#909399` on `#fff` 為約 3.3:1，未達標）。

**根本原因：** 顏色 token 使用錯誤（#909399 vs #606266）。

**建議修正：** 將說明文字顏色改為 `#606266`。

### P-08 — CAPI 儲存後 SaveBtn 消失，無法更新已儲存的 Token

**嚴重程度：** 🔵 Enhancement
**影響任務：** T3（Conversion API 設定）
**位置：** `html/行銷追蹤碼管理.html:648`

```jsx
{!tools.capi.tokenSaved && <SaveBtn onClick={saveCAPI} />}
```

**問題描述：** 當 CAPI Token 已儲存後，「儲存此設定」按鈕消失。用戶若想更新 Token 必須先點「重新設定」清空欄位後才能看到儲存按鈕。這個流程是正確的（避免誤觸），但 UX Spec 未明確規定此行為。建議在 UX Spec 中補充確認此互動設計，使規格與實作一致。

**建議修正：** 不需修改程式碼；建議在 UX Spec Screen 4 「已儲存狀態」下補一行：「儲存按鈕隱藏；用戶需點擊『重新設定』後方顯示儲存按鈕」。

## 五、受測者語錄（模擬非專家用戶）

以下語錄為 AI 根據實際 UI 文字與互動流程推估，代表不熟悉後台系統的用戶在操作時可能說出的話：

1. 「這裡說『填入 ID 後系統將自動安裝追蹤碼』，所以我只要填一串數字就好？不用自己去複製貼上程式碼？這樣比我以前用的容易多了。」（T1，看到說明文字後的正面反應）

2. 「Measurement ID……是什麼？哦，旁邊有個問號，我滑鼠放上去——哦，在 Google Analytics 後台的資料串流那邊。好，那我去找。」（T1，Tooltip 幫助找到 ID 來源）

3. 「我輸入了一個 ID 然後按儲存，它說格式不對，應該要 G- 開頭……但我剛才沒打 G- 啊，我以為只要打數字就好。」（T1，格式驗證錯誤引發的困惑）

4. 「這個 Conversion API……跟 Meta Pixel 有什麼關係？這個卡片是灰色的，我不能點。喔，下面有字說需要先啟用 Meta Pixel。所以 Pixel 是前提？」（T3，CAPI Disabled 狀態引導）

5. 「我已經設定了 FB Pixel，然後點開這個 API 的設定，它叫『FB 伺服器轉換 API』……可是剛才那個卡片叫『FB 轉換 API』，這是同一個東西嗎？我有點不確定。」（T3，名稱不一致造成困惑）

6. 「這個 GTM……我開了之後，出現了一個橘色的警告說我同時啟用了 GA4 和 GTM。它說可能重複觸發？所以我要去 GTM 裡面確認有沒有重複安裝 GA4？好吧，我記下來。」（T4，重複載入警示引發正確行動）

7. 「下面這個統計表格——我看到產品頁瀏覽 1,234 次，這代表有人在看我的產品。啊，旁邊有說這是近 7 天的統計，不是即時的。那我知道追蹤碼有在動了。」（T5，正確解讀觸發次數）

8. 「我把 GA4 關掉，它出來問我確認停用。下面有寫『歷史數據不受影響』——好，那放心了，我的過去的數據還在，只是停止新的追蹤。」（T6，確認框消除疑慮）

9. 「停用之後它說 GA4 已停用，Toast 訊息閃了一下就消失了……我沒看清楚它說什麼，能不能顯示久一點？」（T6，Toast 3 秒消失可能被錯過）
