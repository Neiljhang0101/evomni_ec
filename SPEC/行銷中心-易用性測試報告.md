### 🔴 Critical 問題

**C-1：停用「啟用中」旅程缺少確認 Modal**

- **影響任務：** T3
- **位置：** `html/行銷中心.html`，`JourneyOverviewPage` 元件，Switch onChange handler
- **問題描述：** 當商家將「啟用中」的旅程 Switch 切換為 OFF，系統直接停用並顯示「{旅程名稱} 已停用」Toast，沒有任何確認步驟。非專業使用者可能誤觸，或在不確定後果的情況下停用了正在為客人發訊息的旅程。Spec 明確要求停用已啟用旅程時，需顯示確認 Modal（標題：「確定停用此旅程？」，說明：「停用後，系統將不再觸發此旅程的推播。」，CTA：確定停用 / 取消）。
- **根本原因：** 第一輪修正加入了「未完成設定不能啟用」的防護，但「已啟用 → 停用」的確認 Modal 沒有一起補上。
- **建議修正：**
  ```jsx
  // 停用確認 state
  const [disableJourneyTarget, setDisableJourneyTarget] = React.useState(null);

  // Switch onChange
  onChange={v => {
    if (v && !j.configured) {
      show('請先完成旅程設定，再啟用此旅程', 'warning');
      return;
    }
    if (!v && enabled[j.key]) {
      setDisableJourneyTarget(j); // 開啟確認 Modal
      return;
    }
    setEnabled(e => ({ ...e, [j.key]: v }));
    show(`${j.title} 已啟用`, 'success');
  }}

  // 確認 Modal
  <AdminModal
    open={!!disableJourneyTarget}
    title="確定停用此旅程？"
    confirmLabel="確定停用"
    confirmVariant="danger"
    onConfirm={() => {
      setEnabled(e => ({ ...e, [disableJourneyTarget.key]: false }));
      show(`${disableJourneyTarget.title} 已停用`, 'warning');
      setDisableJourneyTarget(null);
    }}
    onClose={() => setDisableJourneyTarget(null)}
  >
    <p>停用後，系統將不再觸發此旅程的推播。</p>
  </AdminModal>
  ```

### 🟠 Major 問題

**M-1：優惠券新增頁 Tab 2「排除產品」欄位仍缺失**

- **影響任務：** T1（深度設定）
- **位置：** `html/行銷中心.html`，`CouponFormPage`，Tab 2（使用條件），行 355–385
- **問題描述：** Tab 2 已正確實作「最低訂單金額」和「適用產品/分類」，但 Spec 要求的「排除產品」欄位完全缺失。商家若需對部分產品排除優惠（如已打折產品），目前無法在 Prototype 中操作。
- **建議修正：** 在「適用產品/分類」下方加入「排除產品」區塊（可選填），結構與「適用產品」相同（多選 Tag + 搜尋）。

---

**M-2：活動新增頁缺少「活動說明」和「是否啟用」欄位**

- **影響任務：** 深度導覽
- **位置：** `html/行銷中心.html`，`PromotionFormPage`，Tab 1（基本設定），行 540–575
- **問題描述：** Tab Bar 架構已修正，但 Spec 要求的兩個欄位仍缺失：① 「活動說明（前台顯示）」Textarea；② 「是否啟用」Switch（預設 OFF，代表活動建立後不自動上架）。缺少「是否啟用」開關會讓商家無法預先建立草稿而不上架。
- **建議修正：** 在 Tab 1 活動名稱下方補「活動說明」Textarea（選填，helper text：此說明會顯示於顧客前台），活動期間下方補「是否啟用」Switch（預設 OFF）。

---

**M-3：購物車挽回頁「最低購物車金額」欄位缺失**

- **影響任務：** 旅程設定
- **位置：** `html/行銷中心.html`，`CartRecoveryPage`，Tab 1（觸發條件），行 690–720
- **問題描述：** 觸發條件 Tab 已有「觸發等待時間」和「發送冷卻期」，但 Spec 要求的「最低購物車金額」篩選欄位完全缺失。沒有此欄位，商家無法只對高價購物車觸發挽回旅程（常見需求）。
- **建議修正：** 在觸發條件 Tab 加入「最低購物車金額」Number Input，placeholder「不填寫則不限制金額」。

---

**M-4：LINE OA 設定頁「重新設定」缺少確認 Modal**

- **影響任務：** 進階操作
- **位置：** `html/行銷中心.html`，`LineOAPage`，行 850–860
- **問題描述：** Spec 要求點擊「重新設定」時顯示確認 Modal（標題「確定重設 LINE OA 連線？」，說明「重設後，現有 LINE OA 連線設定將被清除，需重新輸入憑證。」），但 Prototype 直接顯示「解除連結」按鈕，按下後無任何確認即清除所有設定。對非技術背景的商家風險極高。
- **建議修正：** 將「解除連結」按鈕改為「重新設定」，點擊後顯示確認 Modal，確認後才清除憑證並切換回未設定狀態。

---

**M-5：LINE OA 設定頁缺少常見錯誤碼說明（401 / 403）**

- **影響任務：** T4（驗證失敗情境）
- **位置：** `html/行銷中心.html`，`LineOAPage`，驗證失敗 Alert 區塊
- **問題描述：** Prototype 只模擬了驗證成功的路徑，沒有實作驗證失敗的 Alert。Spec 要求驗證失敗時顯示錯誤碼說明（401：Channel Secret 錯誤；403：Access Token 權限不足），這對非技術背景使用者至關重要。
- **建議修正：** 加入「模擬失敗」的測試按鈕或驗證失敗 Alert 元件，內容包含錯誤碼 + 修正說明。

---

**M-6：Screen 6 旅程啟用 Switch 位置與 Spec 不符**

- **影響任務：** 旅程設定
- **位置：** `html/行銷中心.html`，`CartRecoveryPage`，行 675–685
- **問題描述：** Spec 要求旅程啟用 Switch 位於 Tab 1「觸發條件」表單內（作為一個正式欄位），Prototype 將它移至頁面頂部的獨立 Status Bar（在 Tab Bar 之外）。雖然功能正確，但非專業使用者在 Tab 1 中找不到「啟用」選項，以為沒有此功能。
- **建議修正：** 在 Tab 1 表單內保留 Switch 作為正式欄位，Status Bar 可選擇性保留或移除。
