## 三、問題清單（依嚴重程度排序）

### 🔴 Critical

#### C-01　Screen 2 Lock Banner CTA 按鈕無反應

- **影響任務：** T2
- **位置：** `html/方案管理與升級流程.html`，LockBannerPage 函式，約第 553、568 行
- **問題描述：** LockBannerPage 中兩個「了解進階電商包」按鈕均未綁定 `onClick`，User 點擊後沒有任何反應，無法從鎖定功能提示進入升級流程。對非專業 User 而言，這等同於「功能壞掉了」。
- **根本原因：** Prototype 展示元件外觀，但未串接 `onLearnMore` prop 或 `onCompare` handler。
- **建議修正：** 在 LockBannerPage 中，為 Variant 1 與 Variant 2 的 CTA 按鈕加上 `onClick` 回呼，例如：
  ```jsx
  // LockBannerPage 頂部加入 prop 或直接呼叫全域 switchDemo
  <Btn variant="primary" size="sm" onClick={() => switchDemo('compare-plans')}>了解進階電商包</Btn>
  ```
  或者引入 App 層的 `onCompare` 回呼至 LockBannerPage。

### 🟠 Major

#### M-01　Screen 1 進度條百分比語意不清，容易誤讀為「剩餘比例」

- **影響任務：** T1
- **位置：** `html/方案管理與升級流程.html`，PlanStatusPage 函式，約第 404–410 行
- **問題描述：** 進度條標籤為「到期進度」，顯示的百分比是「已過天數佔比」（plan-normal 場景：231 天剩 / 全年，顯示 63%）。非專業 User 很可能解讀為「還剩 63%」，但實際是「已消耗 63%」。在 231 天剩餘的狀態下，視覺上看起來有三分之二已消耗，會讓 User 無故感到緊張。
- **根本原因：** 進度條僅有「到期進度」標籤，缺乏「已用 / 剩餘」明確說明；百分比數字未標注含義。
- **建議修正：** 將進度條副標籤改為 `距到期還剩 {N} 天（佔方案期 {total} 天的 {remaining}%）`，或將進度條改為「剩餘比例」（100 − pct），讓「快滿」等於「快到期」，符合直覺。

#### M-02　Screen 1 plan-critical（5天）場景觸發了應只在 ≤1 天才出現的全域橫幅

- **影響任務：** T1、T3
- **位置：** `html/方案管理與升級流程.html`，PlanStatusPage，約第 371–381 行
- **問題描述：** 全域頂部橫幅（紅色緊急橫條）依規格只應在方案剩餘 ≤1 天時才出現，但 Prototype 以 `planState === 'plan-critical'`（5 天）觸發它。5 天的場景應只顯示卡片內的 `el-alert type="error"`，不應有全域橫幅。目前 plan-critical 同時顯示全域橫幅 + 卡片 alert，行為超出規格，可能讓 User 對緊急程度的認知出現偏差（5 天被設計成比規格更緊急的警示）。
- **根本原因：** Demo 場景只有四種狀態，缺少 ≤1 天的獨立場景，因此把「最接近的危急場景」（5 天）同時承載了兩段規格行為。
- **建議修正：** 新增第 5 個 Demo tab `plan-imminent`（剩 0–1 天），讓 ≤1 天的全域橫幅行為有獨立場景。或在觸發條件中改為 `sc.days <= 1`。

#### M-03　Screen 3 方案 Badge 缺少規格指定的裝飾字元，降低視覺區分度

- **影響任務：** T6
- **位置：** `html/方案管理與升級流程.html`，ComparePlansPage，約第 609、632 行
- **問題描述：** UX Spec 指定 Badge 文字為 `✦ 您目前的方案` 與 `✦ 推薦升級`，Prototype 顯示 `目前的方案` 與 `推薦升級`，缺少 `✦` 裝飾符號。此符號提供視覺重量，強化推薦卡片的差異感，缺少時兩個 Badge 外觀太接近，降低引導效果。此外，完整規格文字為「您目前的方案」，Prototype 為「目前的方案」，多一個「您」字會讓語氣更貼近 User。
- **根本原因：** Badge 文字未完整照 Spec copy 實作。
- **建議修正：** 將兩個 Tag 的 children 改為 `✦ 您目前的方案` 與 `✦ 推薦升級`。

### 🟡 Minor

#### m-01　Screen 4 Checkbox 錯誤狀態缺少視覺高亮，只有文字提示

- **影響任務：** T4
- **位置：** `html/方案管理與升級流程.html`，UpgradeInquiryDialog，約第 744–751 行
- **問題描述：** 送出時未勾選確認框，只出現紅色文字錯誤訊息，Checkbox 本身無紅框或顏色變化。User 視線可能已移到「送出詢問」按鈕，不一定能看到下方的紅色文字。UX Spec 明確要求 `el-checkbox` 在 error 狀態應顯示紅框。
- **建議修正：** 在 Checkbox 的 wrapper div 或 label 加上紅色左邊框，例如：`border: errors.confirmed ? '1px solid #F56C6C' : 'none'`，或在 checkbox 外加一圈紅色 outline。

#### m-02　Screen 2 LockBannerPage「各功能 Banner 文案一覽」使用 Tooltip 文案而非 value-prop 文案

- **影響任務：** T2（展示頁合規性）
- **位置：** `html/方案管理與升級流程.html`，LockBannerPage，約第 580–591 行
- **問題描述：** 底部「各功能 Banner 文案一覽」格子中，每個功能顯示的說明文字用的是 `FEATURES_LOCKED[].tip`（Tooltip 用文案），而非 UX Spec Screen 2 規定的 `value-prop` 文案。例如「自動化行銷旅程」的 Tooltip 文字為「根據顧客行為自動觸發…」，但 Banner value-prop 應為「讓行銷工作自動化、用數據驅動再行銷」。這會讓設計審查者誤以為 Banner 文案已正確。
- **建議修正：** 在 `FEATURES_LOCKED` 資料結構中新增 `valueProp` 欄位，對應 UX Spec 的 value-prop 表，並在 LockBannerPage 的文案一覽格中改用 `f.valueProp`。

#### m-03　Screen 6「查看」按鈕只顯示 Toast，未開啟唯讀詳情 Dialog

- **影響任務：** T5（延伸操作）
- **位置：** `html/方案管理與升級流程.html`，InternalUpgradeListPage，約第 879 行
- **問題描述：** 已完成或取消的紀錄點擊「查看」後，只顯示一條 Toast 提示，而非 UX Spec 要求的唯讀申請詳情 Dialog。對內部人員而言，查看完整申請資料（聯絡電話、備注等）是確認作業的必要步驟。
- **建議修正：** 實作唯讀版 Dialog，重用 `ConfirmActivationDialog` 的商家資訊區塊，所有欄位改為展示模式，移除到期日輸入與確認按鈕。

#### m-04　Screen 6 Empty State 缺少 Headline + Subtext 格式

- **影響任務：** T5（邊緣情境）
- **位置：** `html/方案管理與升級流程.html`，InternalUpgradeListPage，約第 856–860 行
- **問題描述：** 列表為空時，只在 `<td>` 中顯示一行文字「目前沒有升級紀錄」或「找不到符合條件的升級紀錄」，未包含圖示與說明副文。Spec 要求使用 Empty State 元件（Headline + Subtext 分兩行）。篩選空結果也缺少 `請嘗試調整搜尋條件` 副文。
- **建議修正：** 用 SharedUI 的 `EmptyState` 元件取代純文字 `<td>`，帶入 Headline、Subtext。

#### m-05　Screen 5「返回方案狀態頁」外觀為 Button，規格要求為 Link 樣式

- **影響任務：** T2（成功後跳轉）
- **位置：** `html/方案管理與升級流程.html`，SuccessPage，約第 773 行
- **問題描述：** UX Spec 規定此操作使用 `el-link type="primary"`（文字連結），Prototype 使用 `Btn variant="plain"`（外框按鈕）。兩個並排的外框按鈕視覺重量相同，降低了「返回首頁」作為主要行動的視覺優先順序。
- **建議修正：** 將「返回方案狀態頁」改為 `<a>` 或加入 `color: C.blue, fontSize: 14` 的純文字連結樣式，與「返回後台首頁」主要按鈕產生層次差異。

#### m-06　兩個 Dialog 送出中文字使用 `...` 而非標準省略號 `…`

- **影響任務：** T4、T5
- **位置：** `html/方案管理與升級流程.html`，UpgradeInquiryDialog 約第 711 行；ConfirmActivationDialog 約第 925 行
- **問題描述：** UX Spec 指定 Loading 狀態文字為 `送出中…` / `開通中…`（使用 Unicode 省略號 U+2026），Prototype 使用三個英文句點 `...`。視覺上相差甚微但不符合規格，且在不同字型下句點間距會有差異。
- **建議修正：** 將兩處 `送出中...` 和 `開通中...` 改為 `送出中…` 和 `開通中…`。

#### m-07　Screen 1 缺少 31–90 天輕提示的 Demo 場景

- **影響任務：** T1、T3（邊緣情境驗證）
- **位置：** `html/方案管理與升級流程.html`，DEMO_TABS，約第 308–321 行
- **問題描述：** UX Spec 定義了 ≤90 天（且 >30 天）的卡片底部輕提示狀態，但 Demo 只有「正常（231天）」和「30天」兩個相鄰場景，中間的 31–90 天輕提示狀態無法在 Prototype 中驗測。雖然實作程式碼存在（第 411–413 行），但無法在 Demo 中觀察。
- **建議修正：** 在 DEMO_TABS 增加 `{ id: 'plan-mild', label: 'S1 方案狀態（60天）' }`，對應 `days: 60, expiry: '2026/07/14', pct: 83, tokenPct: 40`。

### 🔵 Enhancement

#### E-01　Screen 1 兩張卡片的到期 Alert 可能造成「應該按哪一個？」的困惑

- **位置：** PlanStatusPage，`alertBlock()` 函式被 主機卡片 / AI 卡片 / 電商卡片 各呼叫一次
- **說明：** 到期警示在三個卡片中完全重複相同的文字，非專業 User 可能不確定要對哪一個卡片操作，或誤以為有三個問題需要分別處理。實際上方案到期只需一個統一的 CTA。
- **建議：** 將警示 Alert 抽到頁面頂部統一顯示一次，而非每個卡片重複顯示。

#### E-02　Screen 1 進度條缺少「到期日」的對應視覺錨點

- **位置：** PlanStatusPage，約第 404–413 行
- **說明：** 進度條右側只顯示天數（`{sc.days} 天`），沒有顯示對應的到期日期。User 在看進度條時需要分心往上看卡片資訊中的「到期日」欄。可在進度條右側或下方補上 `到期日：{sc.expiry}` 的小字，減少眼睛跳轉次數。

#### E-03　Loading 與 Error 狀態無法在 Prototype 中驗測

- **位置：** 所有 Screens
- **說明：** UX Spec 為每個 Screen 都定義了 Loading skeleton 與 API 失敗 Error 狀態，但 Prototype 沒有提供對應的 Demo 場景切換。建議在 DemoNav 增加「載入中」與「API 錯誤」兩個通用 Demo tab，方便工程師與 PM 驗收這些邊緣狀態。

## 五、受測者語錄

以下為模擬非專業 User 在實際測試場景中，依據畫面呈現「可能說出」的語錄：

1. （看到方案狀態頁進度條顯示 63%）「這個 63% 是剩 63% 還是用掉 63%？還有 231 天耶，但感覺已經快滿了…」

2. （看到 Lock Banner 後點擊「了解進階電商包」沒反應）「按鈕壞掉嗎？我明明點了。再點一次…還是沒反應。這是要聯絡誰嗎？」

3. （看到主機方案和 AI 方案都出現完全相同的橘色警示框）「奇怪，兩個都壞了嗎？還是說只有一個有問題？我應該點哪一個的續約按鈕？」

4. （升級詢問表單忘記勾選確認框，只看到按鈕旁有紅字）「怎麼一直沒反應…喔等等，有文字嗎？在哪裡？」

5. （看到內部後台的「待處理」排在最上方）「好，這個就是要處理的，直接點吧，很清楚。」

6. （方案比較頁兩個 Badge 外觀很像）「這個灰色的是我現在的方案，這個橘色的是建議換的。嗯，差異還算清楚，就是格子裡功能多很多。」

7. （成功送出詢問後看到兩個外框按鈕並排）「成功了。我要回去哪裡？兩個按鈕感覺都差不多…先按這個試試。」

8. （內部後台「查看」按鈕只出現提示訊息）「這個已開通的我想看一下細節，但點了好像只有一個通知跑出來，然後就沒了，可以看詳情嗎？」
