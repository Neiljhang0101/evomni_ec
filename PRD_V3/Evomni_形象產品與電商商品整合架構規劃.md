## 一、現況盤點：兩個「產品」的差異全景

先把兩個系統攤開來對比，才能做出有依據的決策。

### 1.1 截圖功能逐欄解析

從截圖觀察到的現有 CMS 形象產品功能：

| 功能面向 | 現有 CMS 產品（形象站）| 新電商 商品中心 |
| --- | --- | --- |
| **核心定位** | 品牌展示 + 詢問單（Lead Generation）| 商品交易 + 購物流程（Conversion）|
| **頁面設計** | ✅ Grape.js 完整頁面設計器，每個產品都是獨立 Landing Page | ❌ 標準商品頁模板，不支援自由排版 |
| **圖片** | 最多 **99 張**，20MB/張，另有列表圖片（縮圖） | 最多 **10 張**，10MB/張（媒體庫 API 自動 WebP）|
| **語言** | ✅ 繁中 / EN 雙語系（名稱、內容均可雙語）| 繁中為主（多語系 TBD）|
| **分類** | 單一分類系統 | 三層分類系統（根 → 子 → 孫）|
| **價格** | 原價 + 售價（**純展示用**，非交易價）| 原價 + 售價 + 特價 + 成本價（**交易用**，必填）|
| **庫存 / SKU** | ❌ 無 | ✅ SKU 矩陣（最多 2 層，100 組）|
| **相關產品** | ✅ AI 自動推薦 + 手選（最多 5 個）；**不允許跨系統推薦電商商品（決議 18-2）** | TBD（本次商品中心 PRD 未涵蓋）|
| **購物車** | ❌ 無 | ✅ 加入購物車 → 結帳流程 |
| **詢問單** | ✅ 加入詢問單 CTA | ❌ 無 |
| **SEO** | meta 標題/關鍵字/描述 + og:meta（含社群分享圖）+ robots | meta + og:meta + **Product Schema**（JSON-LD，Google 購物需要）|
| **狀態管理** | 公開 / 未公開 / 指定公開時間 | 已上架 / 已下架 / 草稿 / 已排程 |
| **前台路由啟用方式** | ⚠️ 頁面**必須加入選單管理**才有前台 URL（內容創建與導覽分離架構，詳見 WTO 選單架構規格） | ⚠️ **已上架即自動啟用前台 URL**（`/products/{slug}`），不需加入選單管理 |
| **已下架的 URL 行為** | 從選單移除後前台 URL 即失效（不可用網址進入） | URL 保留頁面，顯示「此商品目前無法購買」，不可加入購物車；SEO 權重保留不中斷 |
| **站內搜尋** | ✅ 自訂搜尋關鍵字欄位 | 商品名稱 + SKU 編號 |
| **列表圖片** | ✅ 獨立列表縮圖（與主圖分開）| 第一張主圖自動作為列表縮圖 |
| **Analytics** | 頁面瀏覽 + 詢問單轉換率 | 商品 PV/UV + 加購 + 結帳 + 購買轉換漏斗 |
| **資料體積** | 輕量（JSON + 媒體庫圖片 CDN）| 中等（含 SKU 矩陣、庫存異動 Log）|

### 1.2 結論：這是兩種完全不同的產品實體

CMS 產品 = **「品牌敘事頁面」**，以內容為核心，Grape.js 是靈魂，價格只是參考資訊。

電商商品 = **「交易資料實體」**，以數據為核心，SKU + 庫存 + 金物流串接是骨幹，頁面設計反而是次要的。

這兩者不應該合併成一個實體，就像文章（文章管理模組）和訂單（Part 3）不應該合併一樣。

## 二、客戶情境分析

### Scenario 1｜純形象網站（無電商）
- 只用 CMS 產品，完整 Grape.js 自由設計
- 詢問單作為 Lead Gen 主力
- 不開通電商模組 → 後台看不到商品中心
- **影響：零，現有流程不變**

### Scenario 2｜純電商（無形象站）
- 只用電商商品中心
- 不需要 Grape.js 頁面設計
- 前台商品頁使用標準模板
- **影響：零，兩系統完全不干涉**

### Scenario 3｜同時開案（形象 + 電商）⭐ 最複雜
- 後台同時看到 CMS「產品」和電商「商品中心」
- 商家可能需要同一個品項在兩邊分別維護
- 必須有清楚的導覽分組，避免混淆操作
- 需要提供「推送橋接」讓資料不用重複輸入

### Scenario 4｜形象站 → 升級電商 ⭐ 最常見的升級路徑
- 原本有 CMS 產品資料（名稱、圖片、介紹、分類）
- 升級後需要把 CMS 產品「轉換」為電商商品
- 不能要求商家重新輸入所有資料
- 必須有「一鍵推送至電商」的橋接機制

## 三、整合架構建議

### 3.1 核心結論：分離架構 + 橋接推送

**最佳解：兩個系統保持獨立，透過橋接機制打通升級路徑，共用媒體庫資源。**

不建議合併的理由：
1. **資料結構根本不同**：Grape.js JSON content 和 SKU JSON 根本沒辦法放在同一張表
2. **業務邏輯衝突**：CMS 的「售價」是展示文字；電商的「售價」是實際交易金額，不能共用
3. **頁面渲染邏輯不同**：形象站需要 Grape.js 渲染 Custom HTML；電商站需要 Product Schema JSON-LD + 標準化 UI 組件
4. **Analytics 分開**：形象站追內容閱讀 + 詢問；電商追轉換漏斗 + GMV，混在一起數據會污染
5. **站台分離**：兩個域名意味著兩個不同的 Sitemap、不同的 SEO 策略、不同的前台渲染

### 3.3 橋接推送機制（CMS → 電商）

這是 Scenario 4「升級電商」客戶最需要的功能。

**觸發入口：**
1. **CMS 產品列表頁** — 每個產品列右側操作欄增加「推送至電商」Button（僅開通電商模組的客戶可見）
2. **CMS 產品編輯頁** — 右側輔助欄「進階操作」區域新增「推送至電商」Button
3. **電商商品建立頁** — 在「新增商品」時提供「從形象產品匯入」快速選擇入口

**推送資料對應表：**

| CMS 產品欄位 | → 推送至 | 電商商品欄位 | 說明 |
| --- | --- | --- | --- |
| `name.zh` | → | `name` | 僅帶入繁中版本 |
| 主圖（媒體庫圖片 ID）| → | 主圖（同一 媒體庫圖片 ID）| 共用同一張圖，不複製 |
| 附圖（媒體庫圖片 ID 列表）| → | 附圖（前 10 張）| 超出 10 張則截斷，提示商家 |
| `display_price`（原價）| → | `price`（售價）| 預填但需商家確認；成本價/特價留空 |
| `category_id` | → | **不推送** | 分類系統不同，需商家手動選擇電商分類 |
| `grape_js_content` | → | **不推送**（見 §3.4）| 頁面設計不對應 |
| `cms_product_id` | → | `cms_product_id`（記錄來源）| 橋接欄位，記錄推送來源 |

**推送後行為：**
- 推送後兩邊**完全獨立**，修改 CMS 產品不會影響電商商品（反之亦然）
- 唯一連動：若 CMS 產品刪除，電商商品的 `cms_product_id` 欄位設為 NULL（不刪除電商商品）
- 一個 CMS 產品只能推送一次（防止重複建立電商商品）；若已推送，操作按鈕改為「查看電商商品」

**推送流程 UI：**

```
[CMS 產品頁 → 點擊「推送至電商」]

Step 1 / 2：確認推送資料
─────────────────────────────────────────────────────
  ✅ 產品名稱：「2026 春夏針織毛衣」
  ✅ 主圖：1 張（共用 媒體庫，不重複上傳）
  ⚠️ 附圖：已有 12 張，電商商品最多 10 張，
         超出的 2 張將不帶入，可於電商商品頁手動調整
  ✅ 參考價格：發信系統$ 1,500（將帶入電商售價，請確認後調整）
  ❌ 分類：需在電商系統手動選擇電商分類
  ❌ 頁面設計：頁面設計不適用於電商商品頁（請見說明）

Step 2 / 2：完成後需要補填的資料
─────────────────────────────────────────────────────
  ⚠️ 庫存數量（必填）
  ⚠️ 是否啟用多規格（尺寸/顏色等）
  ⚠️ 電商商品分類（必填）

[取消]  [確認推送，前往電商商品設定]
─────────────────────────────────────────────────────
```

### 3.4 Grape.js 頁面設計與電商商品詳情頁的關係

這是整個整合中最微妙的地方，需要明確決策。

**問題：** CMS 產品有 Grape.js 做出精美的 Landing Page 效果。電商商品頁是標準模板（大圖 + 規格選擇 + 加入購物車）。兩者能接？

**選項 A：完全分開（✅ 定案採用，決議 18-1）**
- 電商商品詳情頁使用標準電商模板，有商品圖 Carousel、規格選擇、加入購物車、商品描述（富文本）
- CMS 形象頁面保持 Grape.js 自由設計
- 兩站互相連結：電商商品頁底部顯示「**查看品牌故事 →**」連結，指向形象站對應產品頁
  - **顯示條件：** 商品的 `cms_product_id` 不為 NULL（即已橋接形象產品）才顯示此連結；純電商商品（`cms_product_id IS NULL`）不顯示
  - **連結用途：** SEO 內鏈串聯；引導消費者深入了解品牌故事
- **優點：** 各自職責清晰；電商頁 SEO 有 Product Schema；形象頁品牌故事完整

**選項 B：電商商品詳情頁內嵌 Grape.js 內容**
- 電商商品頁上半部是標準 UI（圖片/規格/購買按鈕），下半部是從 CMS 拉取的 Grape.js 內容作為「品牌故事區」
- **優點：** 一頁到底的豐富體驗
- **缺點：** 技術複雜（跨域資料拉取）；Grape.js 渲染在電商環境可能有 CSS 衝突；Product Schema 會被 Grape.js HTML 污染

**建議：選項 A（完全分開），加上兩站互連連結**

原因：電商商品頁的 Product Schema（`@type: Product`）和結構化 JSON-LD 是 Google 購物廣告的基礎，若被 Grape.js 的自由 HTML 干擾，會直接影響廣告成效和 SEO。這個技術代價太高。

**導覽分組設計規範：**
- 形象站區段和電商區段以「分隔線 + 群組標籤」明確視覺區分
- 分隔線顏色：`#DCDFE6`；群組標籤文字：`11px`，`#909399`，uppercase
- 純形象站客戶（未開通電商）：看不到電商區段，選單整潔
- 純電商客戶（無形象站）：看不到形象站「產品」等選單

**文章選單的動態規則（重要）：**
- 純形象站：選單只顯示一個「📝 文章」，所有文章預設屬於形象站
- 純電商：選單只顯示一個「📝 文章」，所有文章預設屬於電商站
- 同時開通兩站：出現「📝 文章（形象站）」與「📝 文章（電商站）」兩個入口，各自篩選

### 3.6 文章管理跨站架構規格

電商站有自己的文章需求，與形象站文章的性質不同：

| 用途 | 形象站文章 | 電商站文章 |
| --- | --- | --- |
| 典型內容 | 品牌故事、新品介紹、活動公告 | 購物須知、退換貨政策、常見問題、電商部落格 |
| 前台渲染位置 | 形象站域名（`brand.com/articles/xxx`）| 電商站域名（`shop.brand.com/articles/xxx`）|
| SEO Schema | `Article` / `BlogPosting` | `Article`（電商站 sitemap）|

**資料庫欄位：** `articles` 表新增 `display_scope ENUM('cms', 'ecommerce', 'both') DEFAULT 'cms'`

| 值 | 說明 |
| --- | --- |
| `cms` | 只在形象站前台顯示 |
| `ecommerce` | 只在電商站前台顯示 |
| `both` | 兩站皆顯示（適合跨站活動公告等）|

**後台文章建立頁「適用站台」欄位（僅兩站都開通時才顯示）：**

```
適用站台
● 形象站    ○ 電商站    ○ 兩站皆顯示
```

**「同一篇文章發布到兩站」的處理邏輯：**
- 商家選擇「兩站皆顯示」時，系統以**單一筆記錄**儲存（`display_scope = 'both'`）
- 不複製記錄，兩站前台各自讀取並渲染——修改一次，兩站同步更新
- 兩個後台文章入口皆可進入同一篇文章的編輯頁
- 「兩站皆顯示」文章的 canonical URL：以建立時所在入口的站台為主，另一站設 `rel=canonical` 指向主站，避免 Google 判定重複內容

**各 Scenario 文章管理行為：**

| Scenario | 後台文章入口 | 預設 display_scope | 可否選「兩站皆顯示」|
| --- | --- | --- | --- |
| 純形象站 | 單一「文章」| `cms` | ❌ |
| 純電商 | 單一「文章」| `ecommerce` | ❌ |
| 同時開通兩站 | 形象站文章 ＋ 電商站文章兩個入口 | 依進入的入口決定 | ✅ |

### 3.7 媒體庫共用策略

兩站都透過 媒體庫管理圖片，這是天然的共用優勢，應該最大化利用。

**共用規則：**
- 商家在形象站上傳的商品圖 → 媒體庫 存儲一份
- 推送至電商時，電商商品直接引用同一個 媒體庫圖片 ID → **不重複上傳、不佔用額外儲存**
- 商家在電商後台「移除」一張圖片，只是解除關聯（`product_images` 表刪除紀錄），不刪除 媒體庫 的原始檔案
- 媒體庫端統一管理刪除權限：在 媒體庫後台，若一張圖片被 CMS 產品或電商商品任一方引用，則不允許從 媒體庫 刪除（提示：「此圖片正被 {N} 個頁面使用，請先移除使用後再刪除」）

### 3.8 Analytics 資料分離策略

**形象站 Analytics（數據中心 的「內容分析」區段）：**
- 指標：產品頁 PV / UV、平均停留時間、詢問單提交率
- 維度：產品、分類、流量來源（UTM）
- 工具：Google Analytics 4（形象站專屬 GA4 ID）

**電商 Analytics（數據中心 的「電商分析」區段）：**
- 指標：商品頁 PV、加購率、結帳率、購買率、GMV
- 維度：商品、分類、廣告來源（UTM）、會員分群
- 工具：GA4（電商專屬 GA4 ID）+ Facebook Pixel + LINE Tag

**兩站共用 Analytics（數據中心 的「整合看板」）：**
- 同時有兩站的客戶可以看到「品牌漏斗」：形象站詢問 → 電商下單的轉換路徑
- 需在電商訂單的 `utm_params` 記錄 `origin_domain`，區分流量是否來自形象站

## 六、SEO 策略分工

兩站各自有完整的 SEO 設定，但策略不同：

| SEO 面向 | 形象站（CMS 產品）| 電商站（電商商品）|
| --- | --- | --- |
| **頁面目的** | 品牌認知、詢問轉換 | 交易轉換、Google 購物廣告 |
| **Meta 標題** | 品牌調性文案（如「2026 春夏針織毛衣 — 品牌故事」）| 交易導向（如「2026 春夏針織毛衣 發信系統$1,490 現貨」）|
| **Schema Type** | `Article` 或 `WebPage` | `Product`（含 `Offer`、`AggregateRating`）|
| **Canonical** | `https://brand.com/products/xxx` | `https://shop.brand.com/products/xxx` |
| **Sitemap** | 形象站獨立 sitemap.xml | 電商站獨立 sitemap.xml |
| **og:type** | `article` | `product` |
| **跨站連結** | 可選：「立即購買」連結指向電商站 | 可選：「了解更多品牌故事」連結指向形象站 |

> ⚠️ **重要**：兩站之間不設 canonical 互指（會讓 Google 混亂），各自保持獨立的 canonical URL。

## 八、給工程師的實作摘要

**橋接機制最小可行實作（Phase 1）：**

1. `products` 表新增 `cms_product_id BIGINT UNSIGNED NULL` 欄位（外鍵指向 `cms_products.id`，`ON DELETE SET NULL`）
2. CMS 產品頁 API 新增一個 `POST /api/v1/cms-products/{id}/push-to-ec` Endpoint
3. Endpoint 邏輯：
   - 讀取 CMS 產品基本資料
   - 建立 `products` 草稿記錄（`status = draft`）
   - 複製名稱、媒體庫圖片 ID 列表（前 10 張）、參考價格（帶入 `price`）
   - 設定 `cms_product_id`
   - 回傳新建立的電商商品 ID → 前台跳轉到電商商品編輯頁
4. 在 CMS 產品編輯頁右側欄和列表頁加上「推送至電商」UI（僅開通電商的商家可見）

**一個 CMS 產品只能推送一次的防呆：**

```sql
-- 在推送前先查詢是否已有推送
SELECT id FROM products
WHERE cms_product_id = :cms_product_id
LIMIT 1;

-- 若已存在，回傳現有電商商品 ID 並提示「已推送」
-- 若不存在，執行建立流程
```
