## 2. 產品背景與目標

### 2.1 背景與舊系統問題

Evomni 電商系統需要一套完整的前台會員登入流程。舊 CMS 的登入頁存在以下已知問題，本 PRD 一次修正：

| # | 舊系統問題 | 本 PRD 解法 |
|---|-----------|------------|
| 1 | 登入以 Modal 呈現，行動裝置體驗差、難以深度連結 | 改為獨立全頁（`/login`），支援 `redirect_url` 參數 |
| 2 | 登入失敗只顯示「帳號或密碼錯誤」，未驗證帳號也同一錯誤文案 | 細分錯誤類型，各有專屬文案與行動指引 |
| 3 | 無暴力破解防護（無 CAPTCHA、無速率限制）| 三層防護：前端節流 + IP 速率限制 + 動態 CAPTCHA |
| 4 | 社群登入按鈕樣式不一致、無載入狀態 | 統一品牌樣式 + loading spinner 狀態 |
| 5 | 密碼重設連結無到期提示，過期後錯誤訊息不友善 | 明確標示 24 小時有效、過期後顯示引導重新申請 |
| 6 | 登入後有時停在登入頁而非原目標頁 | 明確 `redirect_url` 機制，登入成功後回導 |
| 7 | 無帳號鎖定機制，暴力破解沒有後手防線 | 兩層鎖定：5次失敗→15分鐘暫鎖；第6次→系統自動停用（詳見 §5.4）|
| 8 | 人工審核模式與 Email 驗證模式提示語混用 | 依驗證模式顯示不同錯誤文案與行動指引 |

### 2.2 設計目標

- 流程清晰：消費者無需指引即可完成登入、忘記密碼、驗證帳號
- 安全可靠：三層安全機制 + 兩層帳號鎖定，防止帳號被暴力破解
- 響應式：桌面（1280px+）與行動裝置（375px+）均最佳化
- 統一品牌：主色調由商家「全站主題色」設定控制，元件樣式統一

## 5. 功能需求

### 5.1 登入方式

系統支援三種登入方式，商家可在後台停用部分方式：

| 方式 | v1.0 狀態 | 後台控制 |
|------|---------|---------|
| Email + 密碼 | ✅ 執行 | 可關閉（需至少一種方式啟用） |
| LINE 登入 | ✅ 執行（需設定 App 憑證）| 獨立 Toggle |
| Google 登入 | ✅ 執行（需設定 App 憑證）| 獨立 Toggle |
| Facebook 登入 | 📋 **後續規劃（v1.0 不執行）**：App 審核流程長、台灣使用率下滑，列 Phase 2；規格已定義，開發時可直接接續 | 獨立 Toggle（Phase 2 實作）|

**Social-only 風險確認（12-4 定案）：**

當商家在後台後台將「Email + 密碼登入」關閉、僅保留社群登入（Social-only 模式）時，設定頁面需顯示風險確認 checkbox，商家必須勾選後才能儲存此設定：

```
⚠️ 停用 Email + 密碼登入的風險提醒

勾選前請確認：
□ 我了解停用後，若消費者的 LINE/Google 帳號失效或被取消授權，
  將無法以任何方式登入，需聯絡客服處理
□ 我了解部分消費者無法或不願使用社群帳號登入（隱私考量）
  
✅ 我已了解上述風險，確認停用 Email + 密碼登入方式
```

- 商家未勾選確認 checkbox 時，「儲存」按鈕 Disabled
- 此確認為一次性（同次儲存），非永久性記憶；每次欲切換至 Social-only 時均需重新確認

### 5.2 完整錯誤情境與文案

| 情境 | 驗證模式 | 錯誤文案 | 行動指引 |
|------|---------|---------|---------|
| 帳號或密碼錯誤 | 任何 | 帳號或密碼錯誤，請確認後重試 | 提供「忘記密碼？」連結 |
| 帳號不存在 | 任何 | **帳號或密碼錯誤，請確認後重試**（12-1 定案：與密碼錯誤使用相同文案，防止帳號列舉攻擊；不再顯示「查無此帳號」）| 提供「忘記密碼？」連結 |
| 帳號尚未驗證 | **Email 驗證模式** | 此帳號尚未啟用。請至信箱收取驗證信，或 [重發驗證信] | 連結至 `/resend-verification?email=xxx` |
| 帳號尚未驗證 | **人工審核模式** | 此帳號尚未啟用。請等待網站管理員審核。 | 無連結（需由管理員主動審核，無重發選項）|
| **帳號鎖定中（暫鎖 15 分鐘）** | 任何 | 帳號因登入嘗試次數過多，已暫時鎖定 15 分鐘。您可 [重設密碼] 立即解鎖，或稍後再試。 | 連結至 `/forgot-password`；顯示剩餘鎖定時間倒數 |
| 帳號已被停用（手動）| 任何 | 此帳號已被停用，如有疑問請聯繫客服 | 無連結（避免帳號列舉）|
| **帳號已被停用（系統自動）**| 任何 | 此帳號因異常登入行為已被系統停用，請聯繫網站管理員。 | 無連結（需由管理員介入解除）|
| 速率限制（IP 層）| 任何 | 操作頻率過高，請稍候 10 分鐘後再試 | 顯示剩餘冷卻時間倒數（選填）|
| 社群登入取消 | — | 已取消 LINE / Google / Facebook 登入 | 返回登入頁 |
| 社群登入失敗 | — | 社群帳號驗證失敗，請重試或使用 Email 登入 | 提供 Email 登入區塊 |

> **辨別手動停用 vs 系統自動停用：** `members.status = 'disabled'` 時查 `member_logs` 最近一筆 action：`admin_disabled`（手動）或 `system_auto_suspended`（自動）→ 顯示對應文案。

### 5.3 安全機制

#### 「記住我」功能（12-2 定案）

| 選項 | Session 有效期 |
|------|--------------|
| 未勾選「記住我」（預設）| 7 天（從登入時起算）|
| 勾選「記住我」| 30 天（從登入時起算）|

- 登入頁顯示「記住我」checkbox，預設**未勾選**
- 「記住我」勾選狀態由 HttpOnly Cookie 控制，不儲存於 localStorage
- 7 天 / 30 天到期後，下次開啟需重新登入（不自動延長）
- 在同一裝置上登出後，Cookie 立即清除，不保留「記住我」狀態

---

#### 層次一：前端節流
- 「登入」按鈕點擊後進入 loading 狀態直到 API 回應，防止重複送出

#### 層次二：速率限制（後端實作）

| 限制維度 | 規則 | 超限行為 |
|---------|------|---------|
| 依 IP 位址 | 1 分鐘內最多 20 次請求（成功 + 失敗合計）| 回傳 429，前端顯示通用錯誤 |
| 依帳號（兩層）| 詳見 §5.4 帳號鎖定機制 | 第一層：暫鎖 15 分鐘；第二層：自動停用 |

#### 層次三：動態 CAPTCHA（Google reCAPTCHA v3）

| 觸發條件 | 任一即觸發 |
|---------|-----------|
| 同一 IP，15 分鐘內累計登入失敗 3 次 | ✅ |
| 同一帳號，累計登入失敗 3 次 | ✅ |

- **平常**：reCAPTCHA v3 Invisible 靜默驗證，不干擾使用者
- **觸發後**：升級為 v2 Checkbox（「我不是機器人」勾選框）顯示在登入按鈕上方
- API：login 請求附帶 `captcha_token`；後端驗證失效則拒絕登入

**reCAPTCHA 金鑰管理（12-3 定案）：**
- **v1.0**：reCAPTCHA Site Key / Secret Key 由 **Evomni 平台統一管理**，商家不需自行申請或設定
- **Phase 2**：開放商家在後台設定自己的 reCAPTCHA 金鑰（適用商家有 reCAPTCHA 管理需求的場景）
- 商家不自行設定時，使用 Evomni 預設金鑰，全平台共用

### 5.4 帳號鎖定機制（兩層）

#### 觸發流程

```
密碼輸入錯誤（每次）
     ↓
members.fail_count += 1
     ↓
fail_count 達到 5？
├── 否 → 返回「帳號或密碼錯誤」（若 fail_count ≥ 3 同步觸發 CAPTCHA）
└── 是 → 【第一層鎖定】
          locked_until = NOW() + 15 分鐘
          顯示：「帳號已暫時鎖定 15 分鐘，可[重設密碼]立即解鎖，或稍後再試」
          ↓
          鎖定期間再次嘗試登入（第 6 次失敗）
          ↓
          【第二層：系統自動停用】
          members.status = 'disabled'
          寫入 member_logs（action = 'system_auto_suspended'）
          顯示：「此帳號因異常登入行為已被系統停用，請聯繫網站管理員」
```

#### 狀態欄位說明

| 欄位 | 型態 | 說明 |
|------|------|------|
| `members.fail_count` | `TINYINT UNSIGNED NOT NULL DEFAULT 0` | 連續密碼錯誤次數；成功登入或密碼重設成功後自動歸零 |
| `members.locked_until` | `TIMESTAMP NULL DEFAULT NULL` | 暫鎖解除時間；`NULL` 或過期則無鎖定；密碼重設成功後清除 |

#### 自動解鎖條件

| 解鎖方式 | 說明 |
|---------|------|
| 時間到期 | `locked_until` 過期後自動解鎖，`fail_count` **不歸零**（下次仍從 5 次起算至第 6 次）|
| **密碼重設成功** | 立即解除鎖定（`locked_until = NULL`）並清零計數（`fail_count = 0`）|
| 後台管理員手動解鎖 | 管理員在後台會員頁手動清除鎖定狀態並歸零計數 |

> 注意：時間到期後 fail_count 不歸零，是為了讓「鎖定期間再失敗 1 次 → 自動停用」的邏輯繼續有效。密碼重設成功才視為使用者確認身份，才一併清零。

#### 後台自動停用後的行為

**後端狀態變更：**
- `members.status` 改為 `'disabled'`
- `member_logs` 寫入一筆記錄：`action = 'system_auto_suspended'`，`triggered_by = 'system'`，`note = '登入失敗累計達 6 次（含鎖定期間），系統自動停用'`
- 後台會員列表該筆狀態顯示「停用」，操作 Log 中顯示「系統自動停用」（有別於管理員手動停用的 `admin_disabled`）
- 解除方式：後台管理員手動將 `members.status` 改回 `'active'`，同步清除 `fail_count = 0`、`locked_until = NULL`

**消費者 Email 通知（自動停用時觸發）：**

| 項目 | 內容 |
|------|------|
| 觸發時機 | `members.status` 被寫入 `'disabled'`（`triggered_by = 'system'`）時立即發送 |
| 寄件名稱 | 商家名稱（同其他通知信，非 Evomni）|
| 主旨 | 您的帳號已被暫停 — [商家名稱] |
| 信件內容 | 「您好，由於您的帳號出現異常登入行為，已被系統暫停。如確認是您本人操作，請聯繫我們的客服協助恢復帳號。」+ 客服聯繫資訊（帶入商家設定的客服信箱 / 電話）|
| 發送管道 | Email（不經 LINE / SMS，確保所有消費者都能收到）|
| 通知模板 | 新增至 Part6 §6.12 通知信清單（需連動更新）|

**管理員端（不主動推播）：**
- 後台「會員管理」列表新增篩選項：**「系統自動停用」**，管理員需要時可集中審核
- 不主動發送 Email / 鈴鐺通知給管理員，避免大型商店通知噪音
- 操作 Log 完整保留，管理員進入會員詳情頁可查看停用原因與時間

## 7. 資料與技術規格（建議參考）

### 7.1 相關 API

| 動作 | Method | Path | 說明 |
|------|--------|------|------|
| Email 密碼登入 | POST | `/api/auth/login` | body: email, password, captcha_token（選填）|
| 忘記密碼申請 | POST | `/api/auth/forgot-password` | body: email |
| 密碼重設 | POST | `/api/auth/reset-password` | body: token, password, password_confirmation |
| Email 驗證 | GET | `/api/auth/verify-email?token=xxx` | 驗證 token 並啟用帳號 |
| 重發驗證信 | POST | `/api/auth/resend-verification` | body: email |
| 登出 | POST | `/api/auth/logout` | 清除 Session / Cookie |

### 7.2 安全規格

```php
// 登入 API 後端速率限制（Laravel Throttle 建議設定）
Route::middleware('throttle:20,1')   // 20次/1分鐘，依 IP
     ->post('/auth/login', ...);

// 帳號層級鎖定邏輯（建議在 LoginController 內實作）
// 每次密碼錯誤 fail_count += 1
// fail_count >= 5 且 locked_until = NULL → 設定暫鎖 15 分鐘
// 已在鎖定期間且再次嘗試（fail_count >= 6）→ auto_suspend

// CAPTCHA token 驗證（reCAPTCHA v3）
if ($captchaTriggered) {
    $verified = Http::post('https://www.google.com/recaptcha/api/siteverify', [
        'secret' => config('services.recaptcha.secret'),
        'response' => $request->captcha_token,
    ])->json('success');
    if (!$verified) return response()->json(['error' => 'captcha_failed'], 422);
}
```

### 7.3 Session / Cookie

- 登入後設定 HttpOnly Cookie（`evomni_session`），防止 XSS 竊取
- **Session 有效期（12-2 定案）：**
  - 未勾選「記住我」：7 天（從登入時起算）
  - 勾選「記住我」：30 天（從登入時起算）
  - 到期後自動登出，需重新登入；不自動延長
- `redirect_url` 參數需過濾：只允許同域路徑，防止 Open Redirect 攻擊

### 7.4 密碼重設 Token

- Token 格式：隨機 64 字元，儲存 hash（`password_reset_tokens` 表）
- 有效期：24 小時
- 使用一次即失效（使用後刪除記錄）

### 7.5 members 表新增欄位（建議）

以下兩個欄位需新增至 `members` 資料表，以支援帳號鎖定機制（連動 Part6 §6 DB Schema）：

**資料表：members（新增欄位，連動 Part6 §6 DB Schema）**

| 欄位 | 業務說明 |
| --- | --- |
| fail_count | 連續密碼錯誤次數；成功登入或密碼重設成功後歸零 |
| locked_until | 帳號暫鎖解除時間；無值或已過期則無鎖定；密碼重設成功後清除 |

> 此規格需同步更新 Part6 會員管理 PRD §7 DB Schema 章節。

### 7.6 member_logs 帳號鎖定事件記錄

系統自動停用時，需在 `member_logs`（或等效操作 Log 表）寫入以下記錄：

| 欄位 | 值 |
|------|----|
| `member_id` | 被停用的會員 ID |
| `action` | `'system_auto_suspended'` |
| `triggered_by` | `'system'` |
| `note` | `'登入失敗累計達 6 次（含暫鎖期間），系統自動停用'` |
| `created_at` | 事件發生時間 |

> 後台會員操作 Log 頁需能顯示此事件，且以「系統自動停用」標示（有別於管理員手動停用的「管理員停用」）。
