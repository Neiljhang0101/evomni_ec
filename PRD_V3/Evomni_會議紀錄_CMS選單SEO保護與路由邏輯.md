## 議題背景

WTO 選單管理的設計哲學為「內容創建與導覽結構分離」：CMS 頁面必須加入選單管理才會產生前台路由（URL）。此設計與電商商品的邏輯不同（電商商品上架即自動擁有 URL）。本次會議針對兩個衍生問題進行討論：

1. **SEO 風險**：CMS 頁面從選單移除後 URL 直接 404，已被 Google 收錄的頁面將損失 SEO 積累。是否需要 301 redirect 機制？
2. **雙系統操作不一致**：CMS 頁面需加入選單才有 URL，電商商品上架即有 URL，同一後台存在兩套心智模型，是否需要在架構或 UX 上對齊？

---

## 第一輪：各自表態

**📋 PM Lee：** 兩個問題都不能忽略，但優先序不同。SEO 風險是立即損失，要先處理。發布邏輯不一致長期一定要對齊，兩套心智模型放在同一個後台會讓商家困惑——不是「說明文字」可以解決的，是架構要修正的。

**🖥 Felix：** SEO 部分沒意見，加 redirect table 不是大工程。但「對齊發布邏輯」需要先確認方向——若要讓 CMS 頁面發布就有 URL，會直接打架 WTO 選單管理的設計哲學。Neil 的 spec 明確寫出「內容創建與導覽分離」為核心原則，這件事應該先問 Neil，不是今天可以決定的。

**⚙️ Ray：** 301 redirect 技術上需要 `page_slug_history` 或類似表記錄頁面曾有過的 URL，否則不知道從哪個路徑 redirect 到哪裡。同一頁面可能被加入選單又移除多次，slug 是否固定也需釐清。發布邏輯對齊涉及架構層決策，同意 Felix，不建議在沒有 WTO 共識的情況下動它。

**💼 Nico：** 客戶有反映此問題。有客戶整理選單時把頁面移掉，Google Search Console 跑出一堆 404，他們完全不知道移除選單會讓頁面消失。支持做 redirect，更支持讓操作變得直覺，不讓客戶踩坑。

**📞 Andy（客服）：** 每個月都有「我發布了頁面但找不到」的客服問題，需要解釋「還要去選單管理加一次」，客戶困惑「那我點發布是在幹嘛」。另一個情境是客戶整理選單後幾週才發現 SEO 受影響，這兩個情境都是真實的高頻問題。

---

## 第二輪：交叉質疑

**📋 PM Lee → 🖥 Felix：** 「打架 WTO 架構」——但我說的對齊不是要推翻選單管理，而是「內容沒有加入選單，連 URL 都不存在」這件事本身是否合理？導覽與內容分離沒問題，但內容應該有自己的 URL，只是「不在選單裡」。這兩件事不衝突吧？

**🖥 Felix 回應：** 有道理，但 WTO 目前的 routing 是掛在選單結構上的。頁面 URL slug 不是頁面自己帶的，是選單節點決定的。要讓頁面在不加入選單的情況下也有 URL，等於在頁面 model 上獨立維護 slug 欄位，routing 要從「看選單」改成「看頁面本身的 slug」。這是兩層架構改動，不是加一個欄位而已。

**⚙️ Ray 補充：** 若頁面有自己的 slug，當同一頁面加入兩個不同選單節點、路徑不同時，以哪個 URL 為準？WTO spec 的麵包屑邏輯建立在「URL 來自選單路徑」的前提上，若改成頁面自帶 slug，整個 routing 和麵包屑邏輯都要重寫。

**💼 Nico：** 客戶最痛的是哪一個——「發布了但沒有 URL」還是「移除選單頁面不見了」？

**📞 Andy：** 兩個都有，但「移除選單頁面不見了」比較後知後覺；「發布了但沒有 URL」是每次建新頁都會遇到的高頻問題。

**📋 PM Lee（退讓）：** 若架構改動太大，短期先不動 routing 邏輯。但有一件事要堅持：**移除選單前要有明確警告，告訴商家這個 URL 即將消失**，並提供 301 redirect 設定選項。這不需要動 WTO 架構，對吧 Felix？

**🖥 Felix：** 可以，在移除動作前加 confirmation dialog。但問題是：我們怎麼知道這個頁面有沒有被 Google 收錄？我們沒有接 GSC API。

**⚙️ Ray：** 不需要知道有沒有被收錄。只要頁面曾以「公開發布」狀態存在超過一定時間（如 24 小時），就一律觸發警告。保守處理，漏報總比誤放好。

---

## 第三輪：收斂討論

**📋 PM Lee（提出折衷方案）：** 不動 WTO routing 核心架構，做兩件事：
1. 移除選單時加警告彈窗，商家可選「設定 301 redirect」或「確認移除（了解後果）」
2. CMS 頁面「儲存/發布」旁邊加顯眼 callout，說明「此頁面尚未加入選單，目前前台無法訪問」

短期先解決客服問題和 SEO 保護，架構對齊另外排優先序討論。

**⚙️ Ray（有條件接受）：** 方向可以接受。redirect 技術上用 `page_slug_history` 記錄頁面曾有過的 URL（含透過哪個選單節點產生的路徑），移除時商家選擇設定 redirect 就寫入 `redirects` 表，前台 API routing 時比對此表。**前提：`page_slug_history` schema 設計需先評估多路徑情境，避免同一頁面加入不同選單產生資料混亂。**

**🖥 Felix（接受）：** 警告 dialog 和 callout UI 可以做。callout 要放在顯眼位置，建議 `<el-alert type="warning">` 放在儲存按鈕正上方，顏色夠醒目。「設定 301 redirect」流程要讓商家從現有頁面選擇目的地，不能只丟一個 input 框。

**💼 Nico：** 支持方案，但警告 dialog 要夠嚇人，客戶不會讀文字。建議確認按鈕 disabled 一段時間，強迫至少看到內容。

**📋 PM Lee：** 移除選單不像刪帳號那麼嚴重，不需要輸入關鍵字確認。但可以讓確認按鈕 disabled 1.5 秒，強迫商家至少看一秒鐘。

**📞 Andy：** 新機制上線後，現有已發生的 404 舊頁面能不能處理？能否在後台提供「掃描失效頁面報告」？

**⚙️ Ray：** 需要比對 `page_slug_history`，但若歷史資料本來就沒有記錄，就掃不出來。新機制只對上線後行為有效，舊資料無法回溯。

**📞 Andy：** 了解，這是已知限制，知道如何跟客戶解釋了。

## ✅ 會議決議

**決定事項：**

1. **移除選單節點警告彈窗**：當商家移除「曾以公開發布狀態存在超過 24 小時」的頁面節點時，觸發 `<el-message-box>` 警告彈窗。提供兩個選項：「設定 301 redirect（選擇目的地頁面）」或「確認移除（了解此頁 URL 將失效）」。確認按鈕 disabled 1.5 秒，強制閱讀。

2. **redirect 技術實作**：建立 `page_slug_history` 記錄頁面歷史路徑；建立 `redirects` 表記錄 301 對應；前台 routing API 在路由未命中時比對 `redirects` 表，命中則回傳 301。

3. **CMS 頁面發布狀態 callout**：頁面在「已發布但未加入任何選單」狀態時，後台編輯頁右側欄顯示 `<el-alert type="warning">`，文案：「⚠️ 此頁面尚未加入任何選單，目前前台沒有可訪問的網址。請至 網站設計 > 選單管理 新增此頁面。」頁面加入至少一個已發布選單節點後自動消失。

4. **發布邏輯架構對齊（暫緩）**：CMS 頁面是否應在不加入選單的情況下也擁有獨立 URL，涉及 WTO routing 核心架構，需與 Neil 另開討論，本次不決定。

**執行前提：**
- `page_slug_history` schema 設計需先評估多選單路徑情境（同一頁面加入兩個不同選單節點時 URL 取哪個）
- redirect 目的地選擇器需提供「從現有頁面選擇」的 select，而非純文字 input
- 新機制只對上線後的行為有效，既有 404 頁面無法回溯掃描（已知限制）

**遺留問題（需後續追蹤）：**
- 與 WTO（Neil）確認：CMS 頁面是否可以有獨立 slug，脫離選單節點也能被訪問
- 確認後，視結論決定是否需要調整 routing 架構，並對應修正本 PRD 的 callout 邏輯

## 💡 主持人觀察（Mia）

這次討論最關鍵的分歧不在 SEO redirect 本身（那個收斂得很快），而在於**雙系統發布邏輯的不一致，到底是「UX 引導問題」還是「架構設計問題」**。PM Lee 和 Felix 代表了兩種立場：PM 認為根本原因在架構，Felix 認為架構是別人定的邊界不能輕易動。

這個議題之所以難以快速收斂，是因為 CMS 的選單架構是 WTO 的設計決策，Evomni 電商是另一個產品線，兩邊的設計哲學本來就不同。要真正對齊，必須跨產品線溝通——這不是今天這個會能解決的範疇。

建議下次討論時帶著具體提案給 Neil：**「CMS 頁面是否能在頁面 model 層級保有獨立 slug，選單管理負責導覽結構，但不壟斷 URL 的生成」**。這個提案若能被接受，才有辦法真正消除雙系統不一致。
