## 二、各區塊功能說明

---

### 🏠 儀表板

**包含功能：**
- 今日銷售額、訂單數、新會員數（即時數字）
- 待處理事項快速入口（待審核退換貨 N 筆、發票未開立 N 筆、庫存告警產品 N 筆）
- 最近 7 天訂單趨勢折線圖
- 熱銷產品 Top 5
- 待確認連線異常的串接廠商警示（若有）

**FAE 說明：** Dashboard 是整個系統的「狀態快照」，所有數字都是唯讀查詢，不做寫入操作，效能要求最高（≤ 1.5 秒初始載入）。Dashboard 的每個「待處理」卡片都是跳轉連結，點擊後直接進入對應模組並帶預篩選條件。

---

### 📦 產品中心

**包含功能：**

| 子項目 | 主要功能 |
| --- | --- |
| 產品管理 | 產品 CRUD、圖片（ML 媒體庫）、規格設定、庫存數量、溫層屬性、產品重量、SEO 設定 |
| 產品分類 | 樹狀分類管理、排序、分類圖片 |
| 庫存管理 | 庫存異動記錄查詢（inventory_logs）、低庫存警示設定、庫存調整 |
| └ 貨到通知訂閱（Tab） | 查看各產品的貨到通知訂閱列表、補貨後批次觸發通知 |

**為什麼把「貨到通知」從會員中心移到這裡：**
- 貨到通知的觸發點是「產品補貨（庫存 0 → >0）」，是庫存管理的下游事件
- 操作者需要確認庫存補充後，才去觸發貨到通知，兩個動作應在同一頁面
- 會員管理的使用者視角是「管理人」，不是「庫存作業人員」

---

### 📋 訂單管理

**包含功能（三個 Tab 合為一個主選單項目）：**

| Tab | 主要功能 |
| --- | --- |
| 全部訂單 | 訂單列表、搜尋篩選、訂單詳情、手動建立訂單、批次出貨/匯出 |
| 退換貨申請 | 退換貨列表、審核流程（核准/拒絕/驗收）、退款金額計算、換貨出貨 |
| 電子發票 | 發票列表、手動補開（貨到付款）、折讓/作廢操作、發票狀態查詢 |

**為什麼退換貨和電子發票合併為 Tab 而非獨立選單：**
- 退換貨是訂單的延伸狀態，操作者通常從訂單詳情頁跳轉過來，心智模型是「訂單的一部分」
- 電子發票永遠對應一張訂單，無法脫離訂單獨立查詢，歸屬感明確
- 從 FAE 角度：三個 Tab 共用同一個訂單 `order_id` 作為查詢橋梁，路由設計為 `/orders?tab=returns`，避免重複 fetch member、product 資訊

---

### 👥 會員中心

**包含功能：**

| 子項目 | 方案限制 | 主要功能 |
| --- | --- | --- |
| 顧客管理 | 兩方案 | 會員列表（搜尋、篩選、匯出）、會員詳情頁（訂單/點數/標籤/黑名單歷程）、手動調整點數 |
| 等級設定 | 進階方案 | 最多 5 個等級設定、升等條件、保級規則、升等禮設定 |
| 點數設定 | 兩方案 | 回饋比率、有效期、兌換比率（進階）、到期提醒天數 |
| 分眾標籤 | 兩方案（進階篩選鎖定）| 系統自動標籤說明、自定義標籤 CRUD、進階多條件篩選器（進階方案）|
| 黑名單 | 進階方案 | 黑名單列表、加入/解除操作（含審計 Log） |
| 設定 | 兩方案 | 自動成為會員開關、推薦功能開關；進階方案另可開關生日雙倍點數（見 Part 6 §6.11） |
| 通知信管理 | 兩方案 | 9 種會員 Email 通知的開關與模板編輯（主旨/內文/變數插入/預覽）（見 Part 6 §6.12） |

**設計說明：**「分眾標籤」雖然和「行銷中心」關係密切，但資料的「主人」是會員，篩選條件基於會員行為數據，所以歸屬在會員中心。行銷中心的活動在「選擇受眾」時，透過抽屜（Drawer）嵌入分眾篩選器，不需要切換選單。

---

### 🎯 行銷中心

**包含功能：**

**優惠活動（四種活動類型共用同一個容器頁面，用 `<el-tabs>` 切換）：**

| Tab | 方案 | 說明 |
| --- | --- | --- |
| 優惠券 | 兩方案 | 折扣碼、百分比/固定金額、入會贈券、生日贈券（進階）、滿額贈券（進階）|
| 滿額 / 滿件優惠 | 兩方案 | 滿額折扣、滿件折扣、滿額贈品、滿件贈品、免運活動 |
| 限時折扣 | 進階方案 | 指定產品時間區間打折、產品頁倒數計時設定 |
| 組合優惠 | 進階方案 | 組合加價購、產品組合價、買 X 享 Y |

**自動化旅程（進階方案，每個旅程是獨立的「設定卡片」）：**

| 旅程 | 觸發事件 | 對應 Channel |
| --- | --- | --- |
| 購物車挽回 | 會員購物車閒置 N 小時 | Email + LINE |
| 沉睡喚醒 | 90 天未購買 | Email + LINE |
| 流失挽留 | 180 天未購買 + 高價值客標籤 | Email + LINE |
| 購後推薦 | 訂單完成後 N 天 | Email + LINE |
| 到期提醒 | 點數 / 優惠券即將到期 | Email + LINE |

**一頁式商店（進階方案）：**
- 落地頁（Landing Page）列表管理（CRUD `/lp/{slug}`）
- 各頁的 PV 漏斗統計（點擊率、加購率、結帳轉換率）
- 倒數計時設定、產品選擇

**為什麼「一頁式商店」放在行銷中心：**
- 一頁式商店的核心用途是廣告投放的落地頁，是行銷工具，不是主要的產品陳列頁面
- 操作者通常是行銷人員（非產品管理人員），使用情境是「建立廣告活動時順便建立落地頁」
- FAE：一頁式商店有自己的路由（`/lp/`），只是後台管理入口從行銷中心進

### 📈 數據中心

**包含功能：**

| 子項目 | 資料來源 | 說明 |
| --- | --- | --- |
| 銷售概覽 | 訂單 | GMV、訂單數、客單價、完成率趨勢、付款方式分佈 |
| 產品分析 | 訂單 + 產品 | 熱銷排行、滯銷排行、庫存周轉率、收藏數（進階）|

| 行銷成效 | 行銷活動 + 訂單 | 各優惠券核銷率、自動化旅程開信率/點擊率/轉換率、一頁式商店漏斗 |

**FAE 說明：** 數據中心是「唯讀聚合查詢」，涉及大量跨表 JOIN，建議使用獨立的分析 DB（讀寫分離）或定時快照機制，不應在 OLTP 主庫直接做複雜報表查詢。匯出功能一律異步處理（寄 Email）。

## 三、哪幾個左側功能彼此高度相關

---

### 關聯群組 A：訂單 ↔ 產品（庫存）

**關聯說明：**
訂單的建立會即時扣減 `products.stock`，訂單取消或退換貨驗收通過後觸發庫存回補（`+1`）。產品的溫層屬性決定可選的物流廠商，產品重量影響運費計算。因此兩個選單在「訂單詳情頁」有雙向跳轉需求：訂單詳情頁的產品圖可點擊進入產品管理，產品管理的庫存異動記錄可篩選「來源為訂單」。

**高頻跨導覽情境：**
1. 客服人員在訂單詳情看到產品規格問題 → 點擊跳至產品管理查原始設定
2. 庫存管理員在庫存頁看到低庫存警示 → 點擊查看有多少待出貨訂單

---

### 關聯群組 B：訂單 ↔ 會員

**關聯說明：**
訂單「完成」事件是會員等級升降和點數發放的觸發器；退款確認事件是 `total_spent` 扣回的觸發器。會員詳情頁需要嵌入「訂單記錄 Tab」，訂單詳情頁需要顯示「消費者會員等級」和「點數使用資訊」。

**高頻跨導覽情境：**
1. 商家管理員在訂單詳情發現消費者有疑似惡意退貨記錄 → 跳至會員詳情 → 加入黑名單
2. 會員管理員在會員詳情查消費記錄 → 點擊訂單編號 → 跳至訂單詳情

**FAE 提醒：** 訂單詳情頁面的「消費者資訊」區塊（右上欄）需要透過 `member_id` 查詢會員數據，建議使用 Vuex/Pinia 共享 member store，避免多個頁面重複 fetch 同一會員資料。

---

### 關聯群組 C：會員 ↔ 行銷（最高耦合度）

**關聯說明：**
行銷活動的受眾選擇依賴會員中心的「分眾標籤」和「進階篩選器」；自動化旅程的觸發條件（沉睡客、流失客、點數到期）完全依賴會員中心的標籤計算結果；優惠券的發放對象是特定分眾結果；等級升等會觸發自動發放升等禮優惠券。

**高頻跨導覽情境：**
1. 行銷人員建立「沉睡客喚醒活動」時，需要從分眾篩選器選取受眾（**不跳轉頁面，而是在行銷中心頁面內嵌右側 Drawer，從 Drawer 直接操作分眾篩選**）
2. 商家在會員中心的「分眾篩選」篩完受眾後，點擊「發送優惠券」→ 跳轉至行銷中心的優惠券發送頁並帶入受眾列表

**設計建議（避免反覆跨頁）：**
行銷活動設定中的「選擇受眾」區塊，使用 `<el-drawer>` 右側抽屜嵌入完整的分眾篩選器，讓行銷人員不必離開行銷設定頁面，在 Drawer 內完成篩選後帶回受眾人數和 member_id 列表，免去兩個模組間的頁面切換。

---

### 關聯群組 D：全域設定（金物流串接）↔ 訂單管理

**關聯說明：**
金流廠商的 API 憑證在「全域設定 > 金物流串接 > 金流設定」中填入，實際的付款觸發和 Webhook 接收運作在訂單流程中。當金流串接發生問題（Webhook 失敗、退款 API 異常），操作者需要：
1. 在訂單管理查看「退款異常」訂單
2. 在全域設定 > 金物流串接 > Webhook 管理查看記錄
兩個入口指向同一問題的不同面向。

**設計建議：**
訂單詳情頁的「付款資訊」區塊，若付款/退款狀態異常，顯示快速連結「查看 Webhook 記錄」，點擊後以新分頁開啟全域設定 > 金物流串接 > Webhook 管理，並預帶訂單號過濾條件。

### 關聯群組 E：全域設定（金物流串接 > 物流與運費）↔ 產品中心 ↔ 訂單結帳

**關聯說明（三角形資料依賴）：**

```
全域設定 > 金物流串接 > 物流與運費
  溫層費率規則 ────────┐
  重量費率規則 ────────┤  運費計算
                        ↓
產品中心                訂單結帳
  products.temperature ─→  取最高溫層
  products.weight ──────→  加總重量
                        ↓
                     /api/v1/orders/calculate-shipping
```

產品有自己的「溫層」和「重量」屬性（設定在產品中心），費率規則設定在全域設定 > 金物流串接 > 物流與運費，兩者結合後在結帳 API 中計算最終運費。產品中心的產品設定頁需要有「溫層」和「重量」欄位，並在欄位旁顯示 Tooltip 說明「此欄位影響結帳時的運費計算，詳見全域設定 > 金物流串接 > 物流與運費」。

### 關聯群組 F：行銷中心（自動化旅程）↔ 全域設定（電商進階設定 > LINE OA）

**關聯說明：**
自動化旅程的 LINE 推播依賴 LINE OA Channel Secret 和 Access Token，必須在「全域設定 > 電商進階設定 > LINE OA 設定」中填入並啟用。若未完成 LINE OA 設定，行銷中心的「自動化旅程」頁面中，LINE 推播開關應顯示為鎖定狀態，並提示「請先完成全域設定 > 電商進階設定 > LINE OA 設定」並附上設定頁連結。

**FAE 提醒：** 後端 API 在旅程設定儲存時，需要驗證 `line_oa_settings.is_connected = true`，避免商家啟用 LINE 推播但 API 憑證未設定，導致靜默失敗。

## 五、合併處理建議清單（優先度排序）

| 優先度 | 合併項目 | 合併後位置 | 合併理由 |
| --- | --- | --- | --- |
| 🔴 高 | 退換貨管理 + 訂單管理 | 訂單管理（Tab） | 資料依賴強，操作者相同，無需獨立選單 |
| 🔴 高 | 溫層運費設定 + 重量運費設定 + 物流廠商設定 | 全域設定 > 金物流串接 > 物流與運費 | 同一設定人員操作，邏輯互相依賴，應一次設定完成 |
| 🔴 高 | 電子發票作業介面（訂單管理 Tab）+ 電子發票 API 設定（全域設定 > 金物流串接）| 分開放置，但頁面間互相連結 | 「作業」和「設定」屬不同層次，不應合併但需導航關聯 |
| 🟡 中 | 銀行金流 + 第三方金流 + 電子發票設定 | 全域設定 > 金物流串接 > 金流設定 | 同性質的「憑證設定」，使用者（技術管理員）相同 |
| 🟡 中 | 貨到通知訂閱（從會員中心移至產品中心庫存管理 Tab）| 產品中心 > 庫存管理 | 觸發源是庫存補貨，不是會員行為 |
| 🟡 中 | LINE OA API 設定（全域設定 > 電商進階設定）與旅程 LINE 開關（行銷中心旅程設定頁，Drawer 嵌入）| 分層處理，非合併 | 技術設定和業務設定分離，設計師注意頁面間的狀態同步提示 |
| 🟢 低 | 優惠活動四種類型（優惠券/滿額滿件/限時折扣/組合）| 行銷中心 > 優惠活動（Tab）| 四種優惠在結帳時共用同一個計算引擎，視覺歸類有助商家理解疊加規則 |


**FAE 關鍵設計原則：**
1. **Tab 狀態持久化**：Tab 選中狀態要反映在 URL query param（`?tab=returns`），讓用戶重整頁面或分享連結後仍停在正確 Tab
2. **跨模組快速跳轉**：訂單詳情、會員詳情的相關連結，使用 `router-link` 傳遞預篩選條件（`?filter=order_id:ORD-XXX`），避免用戶到新頁面後還要手動搜尋
3. **Drawer 嵌入而非頁面跳轉**：行銷中心選擇受眾、訂單管理手動建立等跨模組資料選取場景，使用 `<el-drawer>` 右側抽屜，保持主頁面的流程完整性
4. **權限路由守衛**：進階電商包的功能（等級設定、組合優惠、自動化旅程等）在 Vue Router `beforeEach` 中根據 `store.getters.plan` 判斷，Redirect 至升級說明頁而非 404
