### 

### **目標：**

建立一套標準化、可複用的 UI 元件與設計原則，以統一 evomni 後台系統的視覺風格與使用者體驗。此規範旨在加速設計與開發流程、降低維護成本，並確保未來擴充的所有功能模組皆能提供使用者一致且直覺的操作感受。

### **使用者故事（User Story）：**

* **身為** 一位前端工程師，**我想要** 一套定義清晰的 UI 元件庫與使用規範，**以便於** 我能快速、一致地開發新功能介面，無需重複造輪子或猜測設計細節。  
* **身為** 一位 UI/UX 設計師，**我想要** 一個明確的設計系統，**以便於** 在規劃新功能時，能確保其視覺與互動模式符合現有系統的標準，維持品牌一致性。  
* **身為** 一位使用者（客戶或內部員工），**我想要** 一個操作邏輯一致的後台介面，**以便於** 我在不同功能之間切換時，能憑藉直覺完成任務，降低學習成本。

## **使用套件**

* 使用 Element Plus  \- UI Kit 作為基礎元件庫，在這之上進行些許微調以符合WTO的UI風格

  ### **功能需求（Functional Requirements）：**

  #### **A. 整體設計原則 (Global Design Principles)**

1. **佈局 (Layout)**  
   * **主結構：** 採用三段式佈局：  
     * **左側導覽列 (Sidebar):** 寬度固定(240px)，可收合。用於主功能切換。  
     * **頂部標頭 (Header):** 高度固定(48px)，包含麵包屑導覽、常用工具（如語系切換、通知）及使用者資訊。  
     * **主要內容區 (Main Content):** 頁面核心區域，所有功能操作在此發生。內容區應有統一的內邊距（Padding，例如 24px），確保視覺呼吸感。  
   * **間距 (Spacing):**  
     * 採用 8px 為基礎的網格系統。所有元件之間的外邊距 (Margin) 與內邊距 (Padding) 皆應為 8px 的倍數（如 8, 16, 24, 32px），以確保佈局和諧。  
   * **圓角 (Rounded radius):**  預設0，不使用圓角  
2. **色彩 (Color Palette)**  
   * **主色 (Primary):** \#303133 (深灰色) \- 用於主要操作按鈕  
   * **輔助色 (Secondary):** \#409EFF (藍色) \- 用於次要操作按鈕、當前選中項目、焦點狀態等，引導使用者注意力。  
   * **成功色 (Success):** \#67C23A (範例綠色) \- 用於成功提示、啟用的狀態標籤。  
   * **警告色 (Warning):** \#E6A23C (範例黃色) \- 用於需要使用者注意的提示。  
   * **危險色 (Danger):** \#F56C6C (範例紅色) \- 用於刪除操作、錯誤提示。  
   * **文字顏色 (Text):**  
     * 主要文字: \#303133 (深灰色)  
     * 次要文字: \#606266 (中灰色)  
     * 提示/佔位文字: \#909399 (淺灰色)  
   * **邊框/分隔線 (Border/Divider):** \#DCDFE6 (淡灰色)  
   * **背景色 (Background):**  
     * 主背景: \#F5F7FA (極淡灰藍色)  
     * 容器背景: \#FFFFFF (純白)  
3. **字體 (Typography)**  
   * **字型家族:**  Noto Sans TC, sans-serif (優先使用西文，無則 fallback 至繁中黑體)。  
   * **字體大小:**  
     * 頁面標題 (H1): 20px (Bold)  
     * 區塊標題 (H2): 16px (Bold)  
     * 內文/表格文字: 14px (Regular)  
     * 輔助說明文字: 12px (Regular)

   #### **B. UI 元件規格 (UI Component Specifications)**

1. **按鈕 (Button)**  
   * 用途: 觸發操作。  
   * **類型:**  
     * **主要按鈕 (Primary):** 深灰底白字，用於最重要的操作（如：搜尋、儲存、新增）。  
     * **次要按鈕 (Secondary):** 白底灰框，用於次要或非預設操作（如：清除、取消）。  
     * **文字按鈕 (Text):** 純文字，用於表格中的操作（如：編輯、刪除），滑鼠懸停 (hover) 時變色。  
   * **狀態:** Default, Hover (顏色加深或變亮), Active (點擊時), Disabled (灰色且不可點擊)。  
   * 不使用圓角，這與Element Plus 預設不同，要修改  
2. **輸入框 (Input Field)**  
   * **用途:** 供使用者輸入文字內容。  
   * **類型:**  
     * **文字輸入:** 標準單行輸入框。  
     * **搜尋框:** 帶有搜尋圖示的輸入框。  
   * **狀態:** Default, Focus (邊框變藍色), Disabled (淺灰底), Error (邊框變紅色)。  
   * **互動:** 應包含清晰的 Placeholder 提示文字，提示應輸入哪種類型的資訊。  
   * **輔助說明文字 (Helper Text) \-** 這是指出現在輸入框**下方**的簡短文字，它始終可見，是傳達重要規則的最佳位置。

   **用於「解釋規則」與「預防錯誤」**

   **何時使用：**

* **當欄位有明確的輸入限制或規則時：**  
  * **範例：** 密碼欄位下方顯示「長度需介於 8-16 字元，且包含英文與數字」。  
  * **範例：** 使用者名稱欄位下方顯示「只能使用小寫英文與數字，不可使用特殊符號」。  
* **當欄位用途可能不夠直觀，需要額外解釋時：**  
  * **範例：** 文章的「摘要」欄位下方可補充說明：「此內容將會顯示在社群分享預覽中，建議長度為 120 字以內」。這直接關係到 SEO 與使用者體驗，必須清晰傳達。  
* **即時顯示錯誤提示 (Validation Error)：**  
  * **範例：** 當使用者輸入的 Email 格式錯誤時，下方立刻以紅色文字顯示「電子郵件格式不正確」。  
3. **下拉式選單 (Dropdown / Select)**  
   * **用途:** 從預設選項中選擇一項或多項。  
   * **結構:** 包含標籤、當前選值框、下拉箭頭。  
   * **狀態:** Default, Hover, Open (展開列表時), Disabled。  
4. **日期選擇器 (Date Picker)**  
   * **用途:** 選擇單一日期或日期區間。  
   * **結構:** 兩個輸入框分別顯示開始與結束日期，中間有分隔符號，並配有日曆圖示。  
   * **互動:** 點擊後應彈出日曆面板供使用者選擇。  
     * 不論當前是否已選定日期，**日曆面板的預設顯示月份應為「今日」所在的月份**，並高亮標示出「今日」日期。這為使用者提供了一個清晰的操作起點。  
     * 為了大幅提升常用操作的效率，日曆面板的左側或上方應提供**「今日」快捷選項，可快速地選回今日**。  
5. **資料表格 (Data Table)**  
   * **用途:** 結構化地展示大量數據。  
   * **結構:**  
     * **表頭 (Header):** 14px (Bold)，背景色為 \#F5F7FA。  
     * **表格列 (Row):** 滑鼠懸停時應有背景色變化 (\#F5F7FA)。  
     * **Checkbox:** 第一欄用於多選操作。  
     * **操作欄:** 最後一欄固定放置該列的常用操作（如：編輯、刪除）。  
   * **排序:** 支援點擊表頭進行升序/降序排列。  
6. **分頁 (Pagination)**  
   * **用途:** 將大量數據分割成多頁顯示。  
   * **結構:** 應包含總項目數、每頁顯示數量選擇器、頁碼、前往指定頁面輸入框。  
   * **狀態:** 當前頁碼應高亮顯示。在第一頁時「上一頁」按鈕應禁用，在最後一頁時「下一頁」按鈕應禁用。  
7. **狀態標籤 (Tag / Badge)**  
   * **用途:** 顯示狀態或分類。  
   * **樣式:** 帶有圓角的膠囊形狀。  
   * **色彩:** 需與狀態顏色對應。例如，「啟用」使用成功色，「停用」使用灰色。  
8. **麵包屑 (Breadcrumb)**  
   * **用途:** 顯示使用者在網站結構中的當前位置。  
   * **結構:** 使用 / 作為分隔符。最後一項為當前頁面，不可點擊，顏色為主要文字色。前面的項目為可點擊的連結。

   ### **非功能需求（Non-Functional Requirements）：**

* **響應式設計 (Responsiveness):** 所有元件和佈局應能適應不同螢幕尺寸，特別是在寬度小於 1200px 時，應有合理的排版變化。  
* **可及性 (Accessibility):** 所有可互動元件需支援鍵盤操作（如 Tab 鍵切換、Enter 鍵觸發），圖片需有 alt 標籤，顏色對比度需符合 WCAG 2.2 標準。  
* **效能 (Performance):** 元件載入和互動反應需保持流暢。表格在處理大量數據（如超過 1000 筆）時，應考慮使用虛擬滾動 (Virtual Scrolling) 等技術進行優化。

  ### **操作流程（User Flow）：**

* **設計師使用流程:**  
  1. 當需要設計新功能頁面時，參考此文件中的設計原則與元件規格。  
  2. 從共用的 Figma 或 Sketch 元件庫中拖曳所需的標準元件來組合介面。  
  3. 若現有元件不敷使用，需與產品經理及開發團隊討論，決定是否擴充現有元件或新增元件，並更新此規範文件。  
* **工程師使用流程:**  
  1. 根據產品需求，使用前端框架（如 Vue, React）建立的共用元件庫來開發頁面。  
  2. 所有元件的樣式與行為皆已預先定義，只需專注於業務邏輯的實現。  
  3. 確保新開發的頁面在不同裝置上均能正確顯示。

  ### **備註（Optional）：**

* **持續演進的文件:** 此份文件應被視為一份「活文件 (Living Document)」。隨著產品功能的迭代，應持續更新與擴充。  
* **建立程式碼元件庫:** 強烈建議搭配 Storybook 或類似工具，建立一個與此設計規範文件同步的程式碼元件庫。這能讓設計師與工程師在視覺化環境中查看、測試所有元件，確保設計與實作的 100% 一致性。
